# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

DriverVarázsló is a Windows driver-management utility (Hungarian UI/comments) distributed as a single PyInstaller-built `.exe`. It manages third-party/inbox drivers (list, force-delete, backup/restore, offline WIM/ESD extraction), can repair non-booting Windows installs (BCD/bootloader rebuild from WinPE), pauses Windows Update driver pushes, removes ghost devices, runs a one-click "AutoFix" routine that chains a reboot via a scheduled task, generates an HTML hardware report with S.M.A.R.T. data, and launches bundled stability-test tools (FurMark/Prime95/Linpack/HWiNFO/HD Sentinel) with screen-sleep temporarily disabled and their startup dialogs auto-answered. See [README.md](README.md) for the full feature list.

## Architecture

- **`driver_tool.py`** — the entire application (~6700 lines), two API classes:
  - `DriverToolApi` (line ~277) — the GUI backend, exposed to the frontend as `pywebview`'s `js_api`. Nearly every feature is a method on this class.
  - `CliApi` (line ~4833) — a parallel, full-featured mirror of `DriverToolApi` for the text-menu CLI fallback (`run_cli_mode()` ~line 6128, `main_menu()` ~line 6147). When adding a feature to one, the other usually needs the equivalent — this drifts easily (e.g. CliApi previously lacked WU pause/resume and standalone ghost-device deletion, had a materially weaker `autofix()` than the GUI's `run_autofix`, and its offline DISM `/Add-Driver` call was missing `/ForceUnsigned` that the GUI had; all fixed, but this class of gap can reappear — note that CliApi's `autofix()` is *intentionally* a simpler one-shot design with no reboot-chaining, documented in its own printed output, not a bug to "fix" into parity). The Stress-Test feature (below) is the one deliberate, permanent exception: it's GUI-only — CliApi's own help menu just tells the user to relaunch without `--cli` for it, since it depends on `pywebview`'s window plus Win32 UI-automation that has no meaningful CLI equivalent.
  - Section banners (`# ====...`) mark major feature groups within `DriverToolApi`: driver listing, BCD repair, driver deletion, hardware/ghost-device scan, WU driver install, WU pause/management, backup/restore. `grep -n "# ===="` to navigate.
- **`ui.html`** — the entire frontend: plain HTML/CSS/JS (no framework, no build step), dark "glassmorphism" theme. Talks to Python only through `window.pywebview.api.<method>()` (helper: `api()`) and receives async push events via `window.handlePyEvent(payload)`, which Python calls from `DriverToolApi.emit(event, data)` (`driver_tool.py` ~line 337) via `window.evaluate_js(...)`. JS console errors/uncaught exceptions are forwarded back to the Python logger through `api().js_log(...)`.
- **GUI vs CLI dispatch**: `if __name__ == "__main__"` (driver_tool.py ~line 6374) does a single-instance mutex check, then UAC self-elevates if not admin (both the `--cli` and GUI paths call the shared `_relaunch_elevated()` helper, ~line 6396), then branches: `--cli` arg or `check_webview2_runtime()` failing (WebView2 missing/too old, min v109) routes to `run_cli_mode()`; otherwise it creates the `pywebview` window with `DriverToolApi` as `js_api`.
- **Offline mode**: most driver operations check `self.target_os_path` — when set (via `change_target_os`/`apply_target_os`), operations target a foreign Windows volume (e.g. `D:\Windows`, mounted from WinPE or a second OS) using `dism /Image:...` instead of the live `/Online` system. Always handle both branches when touching driver listing/deletion code. DISM calls that parse English text output (driver listing) must pass `/English` on **both** the GUI and CLI paths, or output on a non-English Windows/WinPE silently fails to parse.
- **WinPE awareness**: code checks `os.environ.get('SystemDrive') == 'X:'` to detect running from WinPE/boot media and redirects large temp downloads/extracts to `C:\DV_Temp` instead of the RAM-disk `X:` drive — this redirect must not depend on any other condition (e.g. whether an offline target OS is also selected), since the whole point is that `X:` is a tiny RAM disk.
- **Packaging**: `resource_path()` (~line 164) resolves bundled resources (`ui.html`) via `sys._MEIPASS` when frozen. `BUILD_NUMBER` (top of file) drives the in-app auto-updater (`check_for_updates`, ~line 450): it fetches `driver_tool.py` from `raw.githubusercontent.com`, compares `BUILD_NUMBER`, and if newer, downloads and replaces `dist/DriverVarazslo.exe` — this published exe is intentionally committed to the repo (`git add -f`) so the updater has something to fetch. Never let an automated/manual edit lower the committed `BUILD_NUMBER` — `check_for_updates` only offers an update on strict `new_build > BUILD_NUMBER`, so a regression silently freezes already-updated users (this happened once; see `bump_build.py`). All network downloads (`check_for_updates`, `perform_update`, `_download_stresstools`, `bump_build.py`) use `ssl.create_default_context()` with certificate/hostname verification **on** — never add `check_hostname = False` / `verify_mode = ssl.CERT_NONE` here, these paths download an `.exe` (or other update payload) that gets executed with admin rights. On a *brand-new* Windows install the root-cert store is still sparse (Windows fetches roots on demand, but only schannel clients — browsers/PowerShell/.NET — trigger that; Python's OpenSSL doesn't), so Python gets `CERTIFICATE_VERIFY_FAILED` for `github.com` (Sectigo/USERTrust root) while `raw.githubusercontent.com` (DigiCert root) works — i.e. the updater succeeds on the same fresh machine where the stresstools download fails. `_download_stresstools` therefore falls back, on that specific error only, to a PowerShell `Invoke-WebRequest` download (schannel: certificate verification still fully on, and as a side effect the missing root gets cached into the Windows store, healing future Python downloads on that machine). This fallback is NOT a verification bypass and must not be turned into one.
- **AutoFix resume-after-reboot**: the AutoFix flow (`run_autofix`, ~line 2990) writes a scheduled task pointing back at the exe with `--resume-autofix` / `--resume-step1`, since cleanup steps (Fast Startup off, ghost device removal) require a reboot mid-flow before driver reinstallation can continue. It's a 3-leg chain: Leg A (fresh start) shows the confirm dialog, pauses WU, schedules a task with `--resume-step1`, reboots. Leg B (relaunched by that task — a brand new process) is where ghost-device/driver deletion actually happens, then it schedules `--resume-autofix` and reboots again. Leg C (`--resume-autofix`) does the WU rescan/install, looping (reboot + re-resume) while new drivers keep getting installed. **Every reboot in this chain uses `shutdown /r /t 0 /f`** (the `/f` force-closes blocking apps) — a reboot without `/f` can hang the unattended chain indefinitely. **Any user choice made in the confirm dialog (Leg A) that needs to affect Leg B's behavior cannot be passed as a plain function parameter** — Leg B is a different OS process with no memory of Leg A's call. Thread it through as an extra flag appended to Leg A's scheduled-task `-Argument` string (alongside `--resume-step1`), then parse it back out of `sys.argv` in `__init__` (same pattern as `self.resume_step1`/`self.resume_mode`) so Leg B can read it via `self.<flag>`. See `skip_printer_drivers`/`AUTOFIX_PRINTER_SKIP_CLASSES` for a worked example. The scheduled-task PowerShell command is built via an f-string with single-quoted `-Execute`/`-Argument` values — always pass the exe path/args through the `_ps_quote()` helper (escapes `'` as `''`) before interpolating, or a path containing an apostrophe (e.g. a username) breaks the resume chain.
- **Subprocess-heavy backend**: most real work shells out via `self._run(cmd, **kwargs)` (a logging wrapper around `subprocess.run`, two near-identical copies exist — one per API class) to `dism`, `pnputil`, `bcdboot`/`bootrec`, `takeown`/`icacls`, `powercfg`, and inline PowerShell (`-Command` heredoc strings) for WMI/CIM queries and Windows Update COM API access.
- **Background task concurrency (`DriverToolApi` only)**: long-running tasks run via `self._safe_thread(task, target)` (~line 401), which centralizes two things: (1) a single `self._task_busy` gate — if any `_safe_thread` task is already running, a new one is rejected with a toast instead of starting, because two concurrent tasks would otherwise stomp on shared state (`self._cancel_flag`, `self.hw_updates_pool`); (2) resetting `self._cancel_flag = False` for the new task, done *only after* the busy-check passes (never reset it in the calling method itself, or a rejected/queued call would silently clear a cancel-in-progress on the task that's actually running). `start_hw_scan()` doesn't use `_safe_thread` (it has its own `_hw_scanning` flag plus emits `hw_scan_result` on every exit path so the UI progress bar never gets stuck) but still participates in the same `self._task_busy` gate for the same reason — installing drivers while a hardware scan is rewriting `self.hw_updates_pool` is a data race. Individual stress-tool launches (`start_stress_tool`, below) are the deliberate exception: they bypass `_task_busy` entirely so the user can fire off several tools back-to-back, relying instead on a narrower `self._stresstools_download_lock` around just the shared-ZIP download/extract step. `load_drivers()` also bypasses `_task_busy` (raw `threading.Thread`) since it's a harmless read-only refresh.
- **Stress-test tools — download & power lock** (`STRESS_TOOLS` dict ~line 185, `STRESS_TOOLS_BULK` ~line 199): FurMark/Prime95/Linpack/HWiNFO64/HD Sentinel ship in one externally-hosted `stresstools.zip` (GitHub release asset, not in this repo — whether a given tool actually launches depends on that ZIP's contents, e.g. an HD Sentinel executable must exist in it). `_download_stresstools()` downloads+extracts it once (marker file gates re-download, `self._stresstools_download_lock` serializes concurrent callers so the "start all" button and an individual per-tool click can't corrupt each other's extraction). `start_stress_tests()` (~line 1354, "start all") only launches `STRESS_TOOLS_BULK` — HD Sentinel is excluded on purpose (it's a disk-health *monitor*, not a load generator) and only reachable via `start_stress_tool('hdsentinel')`. Individual per-tool launches (`start_stress_tool`, ~line 1437) deliberately skip the `task_start`/progress-modal system, using only lightweight `toast` events (explicit "no popup, just launch" product requirement). Both call `_lock_power_for_stress()` (~line 615) first, which disables monitor/standby/hibernate timeouts (AC+DC, via `powercfg /setacvalueindex`/`/setdcvalueindex` on `SUB_VIDEO`/`SUB_SLEEP`); the *original* values are queried once (`_query_power_setting`, parses `powercfg /query` output *positionally* by `0x...` hex index rather than by localized label text — the labels are localized, the hex values and their AC-then-DC order are not) and saved to `HKLM\<STRESS_POWER_REG_KEY>` the first time only, then restored and cleared on the *next app startup* (`_restore_power_after_stress()`, called from `__init__`) or when the user presses the red "stop all tests" button — `stop_stress_tests()` (exposed to the UI: a red button in the Stability Test view and another in the stress progress-modal's completed/error state) tree-kills every launched PID recorded in `self._stress_pids` (`taskkill /T` also takes down Linpack's cmd+engine children), then sweeps by image name (`STRESS_KILL_IMAGES`) as a safety net for UAC-`runas` launches with no PID, and finally restores the power settings. It deliberately bypasses `_task_busy` (must work even while a stress task is still waiting/positioning). Other than that manual stop, there is no "test finished" signal to restore on, since the launched tools are independent detached processes.
- **Stress-test tools — unattended input automation**: each tool's startup dialogs are fed automatically, using a different Win32 technique per tool discovered through real hardware testing (do not regress these — see git history for the failed intermediate attempts):
  - *Linpack* (`LINPACK_PROMPT_SCRIPT` + `_build_linpack_console_script` + `_auto_answer_console`) is driven via **`SendInput`-simulated real keystrokes** into its console window — NOT `stdin=subprocess.PIPE` (piping stdin while also passing `CREATE_NEW_CONSOLE` made Linpack fail to start at all). The automation is **prompt-verified**: before every answer, `_read_console_screen()` (AttachConsole + `CONOUT$` + `ReadConsoleOutputCharacterW`) polls the console's visible text until the expected prompt substring actually appears (60s/step) — never type blind on a fixed timer, on a loaded machine the menus appear seconds late and the answers land on the wrong prompt. `Linpack.exe` is a Bat2Exe-wrapped `.bat` (extractable from `%TEMP%\<rnd>.tmp` while running) whose menus read with `choice` (ONE key, no Enter) but whose run-count reads with `set /p` (line + Enter) — hence the per-step needs-Enter flag in `LINPACK_PROMPT_SCRIPT`: sending Enter after a `choice` menu leaves the CR in the console input buffer, the next `set /p` reads it as an empty line, `RUNS` stays undefined, and the batch's `if %RUNS% LSS 1` becomes a syntax error that **silently kills the whole script** (~1s after the RAM answer — this was the long-mysterious field crash, root-caused and fixed with a live repro). The v1.1.8 stress flow is: action menu `2` → RAM option → run count `10000` → threads `Y` → sleep `N` (the app manages power itself) → HWMonitor `N` (HWiNFO already runs) → Enter on `pause`. The RAM option comes from `_pick_linpack_ram_option()` against `LINPACK_RAM_OPTIONS` (menu-number → GB pairs), picking the largest option that fits the machine's *total* RAM (`_get_ram_gb()`, `GlobalMemoryStatusEx` via ctypes, total rounded *up* since an "8GB" stick reports ~7.85GB) **and** the currently *available* RAM minus ~1.5GB headroom — on a 12GB machine running FurMark+Prime95 alongside, the total-only pick (10GB) would page-thrash the whole box. Before each simulated line, code calls `SetForegroundWindow` and then re-reads `GetForegroundWindow()` to confirm the switch actually took effect — Windows silently refuses foreground-stealing from background processes under several conditions, so this is checked explicitly rather than assumed.
  - *FurMark / Prime95 / HWiNFO64* (`STRESS_CLICK_SEQUENCES`, `_auto_click_sequence`) get their native Win32 buttons/radio-buttons pressed via a **`BM_CLICK` message sent with `PostMessageW`** — not coordinate-based mouse-click simulation (immune to window size/position/DPI, no foreground focus needed), and deliberately NOT `SendMessageW`: SendMessage blocks until the button's click handler returns, and FurMark's "GPU stress test" button opens a *modal* CAUTION dialog from its handler, so a SendMessage click never returns until that dialog closes — the "GO" step could never run (field log showed the call stuck for 50s until the user manually closed the window). A sequence step is one label, a list of localized alternatives (HWiNFO's start button varies with OS display language — `['Indítás', 'Start']` — while FurMark/Prime95 stay English regardless of locale), or a dict with `labels` plus options: `skip_if_found` (skip the step without clicking when a *later* step's control is found instead — Prime95's GIMPS welcome dialog only appears on the *very first* launch; once clicked, `prime.txt` gets `StressTester=1` and every later launch opens straight into "Run a Torture Test"), `optional` + `timeout` (the step's dialog may legitimately never appear — HWiNFO sometimes pops one more confirmation dialog *after* Start, so an optional exact-`OK`/`Igen` step waits 20s for it and silently moves on if it doesn't come), and `exact` (full-text match instead of substring — mandatory for short labels like `OK`/`Igen`, where substring matching would false-positive on any Hungarian `...ok`-suffixed text in HWiNFO's sensor list). Label matching goes through `_normalize_ctrl_text()` which strips the Win32 accelerator `&` **and collapses whitespace runs** — real control texts include `Small FFTs (tests L1/L2/L&3 caches, ...)` and `Just  &Stress Testing` (double space!), so a plain substring match can never hit either (each of these alone broke Prime95 automation in the field). Each step is located with `_find_pid_window_with_child_text()` (up to 60s poll), not `_find_window_for_pid()` (which always returns the *largest* visible window for a PID) — a confirmation dialog is normally smaller than the app's main window, so "largest window" logic would miss it entirely. The *last* step of every sequence is additionally verified by `_verify_final_click()`: the final button (HWiNFO "Indítás", Prime95 "OK", FurMark "GO") always dismisses its dialog, so the button must disappear within a few seconds — if it doesn't, the posted click likely got lost (seen in the field on a machine already running the other 3 stress engines at 100%) and it retries, first with another `PostMessageW` and then with `SendMessageTimeoutW` (the old synchronous delivery, but with a bounded 3s wait so a modal-opening button still can't deadlock it). Mid-sequence steps don't need this — the next step's window hunt is their implicit verification.
  - *Any console app* (`_find_console_window_for_pid`, ~line 835, using `AttachConsole`/`GetConsoleWindow`/`FreeConsole`): a console window's true owner is usually a hidden `conhost.exe` process, not the spawned app's own PID, so plain `EnumWindows` + `GetWindowThreadProcessId` (what `_find_window_for_pid` uses) never finds it — confirmed on real hardware (Linpack's window was still "not found" after a 30s wait while FurMark/Prime95/HWiNFO64's GUI windows were all found fine by the PID-based approach). `AttachConsole`/`FreeConsole` mutate process-global state (not per-thread), so every call is serialized through `self._console_attach_lock`.
  - Poll timeouts here are deliberately long (60s) and `start_stress_tests()` launches the 4 tools with a **3-second stagger**, not all at once — launching FurMark+Prime95+Linpack+HWiNFO64 simultaneously (especially alongside an in-flight `dism /Get-Drivers` call, which also fires on startup for the driver-list refresh) can overload a weak machine badly enough that a confirmation dialog takes *closer to a minute* than a few seconds to appear; a real capture showed FurMark's "GO!" button still succeeding at the 56-second mark, well past what used to be a 15s timeout. If you touch these numbers, err long, not short.
  - The three long-standing field failures were root-caused and fixed with live on-hardware repros (2026-07): Prime95 = missing-welcome-dialog skip + `&`-accelerator matching, FurMark = SendMessage modal deadlock, Linpack = wrong answer list for v1.1.8 + stray Enter aborting the batch (details in the bullets above). Each fix was verified by actually launching the tool and confirming the load runs (Prime95 CPU-time delta, FurMark render window with FPS title, Linpack solver output). If a new field report comes in, start from the diagnostics bullet below — and still prefer adding more logging over guessing at another timing fix.
- **Stress-test tools — diagnostics**: because Win32 UI-automation failures are otherwise silent, every call in the automation path (`SendInput`, `SetForegroundWindow`, `AttachConsole`, `PostMessageW`, ...) has its return value checked and logged rather than just invoked — e.g. `_send_unicode_char`/`_send_vk` (~line 803/820) return whether `SendInput` actually injected the event (0 means blocked) instead of assuming success. `_debug_dump_pid_windows(pid, context)` (~line 760) is the go-to failure-diagnosis helper: on any "not found" path it enumerates *every* top-level window for that PID (title/class/visibility/rect) plus every child control (title/class) and dumps it as one `[STRESSTOOLS-DEBUG]`-tagged warning, so a failed match shows what *was* actually there instead of just "not found". Automation background threads are wrapped through `_run_automation_safely()` (inside `_launch_stress_exe`, ~line 1057) so an uncaught exception inside a daemon thread logs a full traceback instead of vanishing silently. When debugging a report from the field, grep the user's `DriverVarázsló_debug.log` for `STRESSTOOLS-DEBUG` first.
- **Stress-test tools — post-launch window layout** (`_position_stress_windows`): runs only after **all automation background threads have finished** (each `_launch_stress_exe` call appends its automation thread into the caller's `thread_sink` list; `start_stress_tests` waits on them with a 240s ceiling, then a further ~10s settle for the final windows — e.g. FurMark's render window appears a few seconds *after* GO is clicked). It used to be a fixed 30s timer, which routinely fired mid-automation: `_minimize_other_windows` minimized the still-unanswered dialogs and FurMark's render window didn't exist yet when positioning ran. Layout is 4 screen quadrants of the work area: FurMark top-left, Prime95 top-right, Linpack bottom-left, HWiNFO bottom-right. The 3 non-FurMark windows are resized to their quadrant; FurMark is *moved* to the top-left corner at native size (`SWP_NOSIZE` — moving doesn't crop the render surface, resizing does: a forced resize just crops the picture, hiding the FPS counter) and pushed to `HWND_BOTTOM`, so exactly its top-left quarter shows in the free quadrant. `_minimize_other_windows()` then minimizes every other visible window — including DriverVarázsló's own window and any leftover per-tool config dialogs — except the 4 just-positioned ones, per explicit user request; expect the app's own progress-modal to vanish into the taskbar along with everything else once this step runs.
- **System report generation** (`generate_system_report`, in `DriverToolApi`, ~line 4410) builds an HTML report via raw f-string concatenation (not a template engine), called synchronously (not via `_safe_thread`). Any value sourced from WMI/CIM or `smartctl` JSON must go through the local `g(dict, key, default)` helper (handles WMI nulls) and `e(value)` helper (HTML-escapes) before interpolation — both are defined inline at the top of the function, not module-level.

## Build / release

There is no test suite and no `requirements.txt`; the only third-party runtime dependency is `pywebview` (imported with a try/except guard at the top of `driver_tool.py` that exits with a clear message if missing).

- **Full release** (build-number bump, PyInstaller build, commit, push): run `rebuild.bat` from the repo root. It calls `bump_build.py` first, which increments `BUILD_NUMBER` from `max(local, GitHub-published)` and keeps `version_info.txt`'s `filevers`/`prodvers`/`FileVersion`/`ProductVersion` in sync with the new build number — never edit `BUILD_NUMBER` by hand and commit a lower value, since that breaks the auto-updater for users already past it.
- **Manual build only** (no commit/push): `python -m PyInstaller --clean DriverVarazslo.spec`. Output is `dist/DriverVarazslo.exe`. The spec embeds `icon_drivervarazslo.ico` and `version_info.txt` (Windows file-version resource — keep this populated and in sync with `BUILD_NUMBER`; an unsigned exe with no/stale version info is far more likely to be flagged/deleted by Defender) and explicitly disables UPX compression for the same Defender-false-positive reason.
- **Run from source**: `python driver_tool.py` (GUI, needs admin — it self-elevates via UAC) or `python driver_tool.py --cli` (text menu).
- `DriverVarazslo.spec`, `version_info.txt`, and `dist/DriverVarazslo.exe` are intentionally tracked in git (not gitignored) — the spec and version-info files are static build inputs `rebuild.bat` depends on existing, and the dist exe is what the in-app updater downloads.
- Debug logging goes to `DriverVarázsló_debug.log` next to the exe (or script when run from source), via a `RotatingFileHandler` (5 MB × 2 backups) — don't swap this back to plain `logging.basicConfig(filename=...)`, the DEBUG-level log (every subprocess's stdout, up to 4000 chars each) grows unbounded otherwise, especially on a long-lived service USB/WinPE install run across many machines.
