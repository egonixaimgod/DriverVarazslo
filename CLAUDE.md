# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

DriverVar├Īzsl├│ is a Windows driver-management utility (Hungarian UI/comments) distributed as a single PyInstaller-built `.exe`. It manages third-party/inbox drivers (list, force-delete, backup/restore, offline WIM/ESD extraction), can repair non-booting Windows installs (BCD/bootloader rebuild from WinPE), pauses Windows Update driver pushes, removes ghost devices, runs a one-click "AutoFix" routine that chains a reboot via a scheduled task, generates an HTML hardware report with S.M.A.R.T. data, and launches bundled stability-test tools (FurMark/Prime95/Linpack/HWiNFO/HD Sentinel) with screen-sleep temporarily disabled and their startup dialogs auto-answered. See [README.md](README.md) for the full feature list.

## Architecture

- **`driver_tool.py`** ŌĆö the entire application (~7600 lines), two API classes:
  - `DriverToolApi` (line ~528) ŌĆö the GUI backend, exposed to the frontend as `pywebview`'s `js_api`. Nearly every feature is a method on this class.
  - `CliApi` (line ~5650) ŌĆö a parallel, full-featured mirror of `DriverToolApi` for the text-menu CLI fallback (`run_cli_mode()` ~line 7033, `main_menu()` ~line 7052). When adding a feature to one, the other usually needs the equivalent ŌĆö this drifts easily (e.g. CliApi previously lacked WU pause/resume and standalone ghost-device deletion, had a materially weaker `autofix()` than the GUI's `run_autofix`, and its offline DISM `/Add-Driver` call was missing `/ForceUnsigned` that the GUI had; all fixed, but this class of gap can reappear ŌĆö note that CliApi's `autofix()` is *intentionally* a simpler one-shot design with no reboot-chaining, documented in its own printed output, not a bug to "fix" into parity). The Stress-Test feature (below) is the one deliberate, permanent exception: it's GUI-only ŌĆö CliApi's own help menu just tells the user to relaunch without `--cli` for it, since it depends on `pywebview`'s window plus Win32 UI-automation that has no meaningful CLI equivalent. The Temp F├Ījlok T├Črl├®se (temp-cleanup) feature sidesteps this drift risk on purpose ŌĆö see its own bullet below: its core logic lives in module-level functions shared by both classes instead of being duplicated.
  - Section banners (`# ====...`) mark major feature groups within `DriverToolApi`: driver listing, BCD repair, driver deletion, hardware/ghost-device scan, WU driver install, WU pause/management, backup/restore. `grep -n "# ===="` to navigate.
- **`ui.html`** ŌĆö the entire frontend: plain HTML/CSS/JS (no framework, no build step), dark "glassmorphism" theme. Talks to Python only through `window.pywebview.api.<method>()` (helper: `api()`) and receives async push events via `window.handlePyEvent(payload)`, which Python calls from `DriverToolApi.emit(event, data)` (`driver_tool.py` ~line 589) via `window.evaluate_js(...)`. JS console errors/uncaught exceptions are forwarded back to the Python logger through `api().js_log(...)`.
- **GUI vs CLI dispatch**: `if __name__ == "__main__"` (driver_tool.py ~line 7289) does a single-instance mutex check, then UAC self-elevates if not admin (both the `--cli` and GUI paths call the shared `_relaunch_elevated()` helper, ~line 7311), then branches: `--cli` arg or `check_webview2_runtime()` failing (WebView2 missing/too old, min v109) routes to `run_cli_mode()`; otherwise it creates the `pywebview` window with `DriverToolApi` as `js_api`.
- **Offline mode**: most driver operations check `self.target_os_path` ŌĆö when set (via `change_target_os`/`apply_target_os`), operations target a foreign Windows volume (e.g. `D:\Windows`, mounted from WinPE or a second OS) using `dism /Image:...` instead of the live `/Online` system. Always handle both branches when touching driver listing/deletion code. DISM calls that parse English text output (driver listing) must pass `/English` on **both** the GUI and CLI paths, or output on a non-English Windows/WinPE silently fails to parse. That text parsing (4 near-identical copies: GUI online/offline, CLI online/offline) keys off substrings like `"Published Name"`/`"Provider Name"`/`"Class Name"` ŌĆö the driver version in real `dism /Get-Drivers` output is a standalone `Version :` line (**not** `"Date and Version"`, which this logic incorrectly assumed for a while and which never matched anything, silently leaving the Version column blank for every driver until field-diagnosed against real DISM output).
- **WinPE awareness**: code checks `os.environ.get('SystemDrive') == 'X:'` to detect running from WinPE/boot media and redirects large temp downloads/extracts to `C:\DV_Temp` instead of the RAM-disk `X:` drive ŌĆö this redirect must not depend on any other condition (e.g. whether an offline target OS is also selected), since the whole point is that `X:` is a tiny RAM disk.
- **Packaging**: `resource_path()` (~line 164) resolves bundled resources (`ui.html`) via `sys._MEIPASS` when frozen. `BUILD_NUMBER` (top of file) drives the in-app auto-updater (`check_for_updates`, ~line 708): it fetches `driver_tool.py` from `raw.githubusercontent.com`, compares `BUILD_NUMBER`, and if newer, downloads and replaces `dist/DriverVarazslo.exe` ŌĆö this published exe is intentionally committed to the repo (`git add -f`) so the updater has something to fetch. `raw.githubusercontent.com` sits behind a Fastly CDN whose edge cache can serve a stale (pre-push) copy of `driver_tool.py` for a few minutes after `rebuild_verzioszam_novelessel_es_github_pushal.bat` pushes ŌĆö the `?t=<timestamp>` query string only defeats client-side caching, not the CDN edge cache, so `check_for_updates` retries up to `UPDATE_CHECK_ATTEMPTS` (3) times with `UPDATE_CHECK_RETRY_SEC` (3s) pauses before giving up; this is a mitigation, not a guarantee ŌĆö testing the updater immediately after a push can still legitimately report "no update" if the CDN lag exceeds the retry window. Never let an automated/manual edit lower the committed `BUILD_NUMBER` ŌĆö `check_for_updates` only offers an update on strict `new_build > BUILD_NUMBER`, so a regression silently freezes already-updated users (this happened once; see `bump_build.py`). All network downloads (`check_for_updates`, `perform_update`, `_download_stresstools`, `bump_build.py`) use `ssl.create_default_context()` with certificate/hostname verification **on** ŌĆö never add `check_hostname = False` / `verify_mode = ssl.CERT_NONE` here, these paths download an `.exe` (or other update payload) that gets executed with admin rights. On a *brand-new* Windows install the root-cert store is still sparse (Windows fetches roots on demand, but only schannel clients ŌĆö browsers/PowerShell/.NET ŌĆö trigger that; Python's OpenSSL doesn't), so Python gets `CERTIFICATE_VERIFY_FAILED` for `github.com` (Sectigo/USERTrust root) while `raw.githubusercontent.com` (DigiCert root) works ŌĆö i.e. the updater succeeds on the same fresh machine where the stresstools download fails. `_download_stresstools` therefore falls back, on that specific error only, to a PowerShell `Invoke-WebRequest` download (schannel: certificate verification still fully on, and as a side effect the missing root gets cached into the Windows store, healing future Python downloads on that machine). This fallback is NOT a verification bypass and must not be turned into one.
- **AutoFix resume-after-reboot**: the AutoFix flow (`run_autofix`, ~line 3716) writes a scheduled task pointing back at the exe with `--resume-autofix` / `--resume-step1`, since cleanup steps (Fast Startup off, ghost device removal) require a reboot mid-flow before driver reinstallation can continue. It's a 3-leg chain: Leg A (fresh start) shows the confirm dialog, pauses WU, schedules a task with `--resume-step1`, reboots. Leg B (relaunched by that task ŌĆö a brand new process) is where ghost-device/driver deletion actually happens, then it schedules `--resume-autofix` and reboots again. Leg C (`--resume-autofix`) does the WU rescan/install, looping (reboot + re-resume) while new drivers keep getting installed. **Every reboot in this chain uses `shutdown /r /t 0 /f`** (the `/f` force-closes blocking apps) ŌĆö a reboot without `/f` can hang the unattended chain indefinitely. **Any user choice made in the confirm dialog (Leg A) that needs to affect Leg B's behavior cannot be passed as a plain function parameter** ŌĆö Leg B is a different OS process with no memory of Leg A's call. Thread it through as an extra flag appended to Leg A's scheduled-task `-Argument` string (alongside `--resume-step1`), then parse it back out of `sys.argv` in `__init__` (same pattern as `self.resume_step1`/`self.resume_mode`) so Leg B can read it via `self.<flag>`. See `skip_printer_drivers`/`AUTOFIX_PRINTER_SKIP_CLASSES` for a worked example. The scheduled-task PowerShell command is built via an f-string with single-quoted `-Execute`/`-Argument` values ŌĆö always pass the exe path/args through the `_ps_quote()` helper (escapes `'` as `''`) before interpolating, or a path containing an apostrophe (e.g. a username) breaks the resume chain.
- **WU driver scan/install ŌĆö shared core (module-level, ~line 185ŌĆō380)**: the PnP device query+filter (`WU_PNP_QUERY_PS` + `_filter_wu_scan_devices`), the WU-resultŌåödevice matching (`_match_wu_updates_to_devices`: HWID *prefix* matching primary ŌĆö substring matching false-positives on short IDs like `usbmmidd` ŌĆö plus titleŌåödevice-name fallback, without which SoftwareComponent packages like the Realtek services never match), and the WUA installer PowerShell script (`_build_wu_install_ps`) each exist in exactly ONE module-level copy, used by all three consumer paths: GUI manual install (`start_hw_scan` pool + `_install_wu_api`, filtering by the selection's exact `UpdateID`s with HWID-prefix fallback), GUI AutoFix (`_scan_and_install_wu_sync`, all matched UpdateIDs per round), and CLI AutoFix (F├üZIS 6, `match_system_devices=True` ŌĆö matching happens inside the script since the CLI does no Python-side pre-filtering). **Never copy this logic back into a class** ŌĆö the duplicated era is what produced the "AutoFix works, manual install silently broken for a whole release" bug (Build ~192). Rules baked into the shared script: downloads are **synchronous `$DL.Download()`** ŌĆö never `BeginDownload($null, $null, $null)`, WUA requires real callback COM objects and null callbacks throw an immediate NullReferenceException before anything downloads (that was the Build-192 regression; hang protection is the Python side's cancel-button `taskkill` instead); selection filtering is by exact `UpdateID` (`$TargetUIDs -contains $U.Identity.UpdateID`) because distinct WU packages can share one hardware ID (Realtek Extension + MEDIA on the same `hdaudio\...` ID), so HWID-only filtering installs more than the user selected (field-seen: 3 selected ŌåÆ 4 installed); and with no filter args the script installs NOTHING (`EMPTY`) ŌĆö there is deliberately no "install everything" default. The script emits a line protocol (`INIT/SEARCH/FOUND/SKIP/TOTAL/DLONE/INSTONE/OK/FAIL/EMPTY/DONE/ERROR`) that all three callers parse ŌĆö if you add a marker, update all three reader loops.
- **Subprocess-heavy backend**: most real work shells out via `self._run(cmd, **kwargs)` (a logging wrapper around `subprocess.run`, two near-identical copies exist ŌĆö one per API class) to `dism`, `pnputil`, `bcdboot`/`bootrec`, `takeown`/`icacls`, `powercfg`, and inline PowerShell (`-Command` heredoc strings) for WMI/CIM queries and Windows Update COM API access. Both `_run` copies default `stdin` to `subprocess.DEVNULL` ŌĆö do not remove this: after the stress-test automation's `AttachConsole`/`FreeConsole` cycles the process's inherited stdin handle goes stale, and inherited-stdin + `capture_output` then fails *every* later command with `[WinError 6]` (field-proven: `taskkill` stopped working right after a stress run).
- **Background task concurrency (`DriverToolApi` only)**: long-running tasks run via `self._safe_thread(task, target)` (~line 659), which centralizes two things: (1) a single `self._task_busy` gate ŌĆö if any `_safe_thread` task is already running, a new one is rejected with a toast instead of starting, because two concurrent tasks would otherwise stomp on shared state (`self._cancel_flag`, `self.hw_updates_pool`); (2) resetting `self._cancel_flag = False` for the new task, done *only after* the busy-check passes (never reset it in the calling method itself, or a rejected/queued call would silently clear a cancel-in-progress on the task that's actually running). `start_hw_scan()` doesn't use `_safe_thread` (it has its own `_hw_scanning` flag plus emits `hw_scan_result` on every exit path so the UI progress bar never gets stuck) but still participates in the same `self._task_busy` gate for the same reason ŌĆö installing drivers while a hardware scan is rewriting `self.hw_updates_pool` is a data race. Individual stress-tool launches (`start_stress_tool`, below) are the deliberate exception: they bypass `_task_busy` entirely so the user can fire off several tools back-to-back, relying instead on a narrower `self._stresstools_download_lock` around just the shared-ZIP download/extract step. `load_drivers()` also bypasses `_task_busy` (raw `threading.Thread`) since it's a harmless read-only refresh.
- **Stress-test tools ŌĆö download & power lock** (`STRESS_TOOLS` dict ~line 185, `STRESS_TOOLS_BULK` ~line 199): FurMark/Prime95/Linpack/HWiNFO64/HD Sentinel ship in one externally-hosted `stresstools.zip` (GitHub release asset, not in this repo ŌĆö whether a given tool actually launches depends on that ZIP's contents, e.g. an HD Sentinel executable must exist in it). `_download_stresstools()` downloads+extracts it once (marker file gates re-download, `self._stresstools_download_lock` serializes concurrent callers so the "start all" button and an individual per-tool click can't corrupt each other's extraction). `start_stress_tests()` (~line 1911, "start all") only launches `STRESS_TOOLS_BULK` ŌĆö HD Sentinel is excluded on purpose (it's a disk-health *monitor*, not a load generator) and only reachable via `start_stress_tool('hdsentinel')`. Individual per-tool launches (`start_stress_tool`, ~line 2019) deliberately skip the `task_start`/progress-modal system, using only lightweight `toast` events (explicit "no popup, just launch" product requirement) ŌĆö and get the exact same dialog automation as the bulk launch (same `console_script`/`click_sequence` wiring), just without the progress modal and without the post-launch window-quadrant layout (that's bulk-only). Both call `_lock_power_for_stress()` (~line 887) first, which disables monitor/standby/hibernate timeouts (AC+DC, via `powercfg /setacvalueindex`/`/setdcvalueindex` on `SUB_VIDEO`/`SUB_SLEEP`); the *original* values are queried once (`_query_power_setting`, parses `powercfg /query` output *positionally* by `0x...` hex index rather than by localized label text ŌĆö the labels are localized, the hex values and their AC-then-DC order are not) and saved to `HKLM\<STRESS_POWER_REG_KEY>` the first time only, then restored and cleared on the *next app startup* (`_restore_power_after_stress()`, called from `__init__`) or when the user presses the red "stop all tests" button ŌĆö `stop_stress_tests()` (exposed to the UI as a red button in the stress progress-modal's completed/error state only ŌĆö deliberately NOT a permanent button in the Stability Test view, per explicit user request) tree-kills every launched PID recorded in `self._stress_pids` (`taskkill /T` also takes down Linpack's cmd+engine children), then sweeps by image name (`STRESS_KILL_IMAGES`) as a safety net for the rare case a launch still has no PID, and finally restores the power settings. It deliberately bypasses `_task_busy` (must work even while a stress task is still waiting/positioning). Other than that manual stop, there is no "test finished" signal to restore on, since the launched tools are independent detached processes.
- **Stress-test tools ŌĆö admin-elevated launches (HWiNFO64)**: `_launch_stress_exe` (~line 1530) first tries a plain `subprocess.Popen`; if that raises `OSError` with `winerror == 740` (`ERROR_ELEVATION_REQUIRED` ŌĆö HWiNFO64.exe's manifest is `requireAdministrator`), it falls back to `ShellExecuteExW` with `SEE_MASK_NOCLOSEPROCESS` (the `_SHELLEXECUTEINFOW` ctypes struct, module-level, near `_Input`) to get a real `hProcess`, then `GetProcessId` on that handle for the actual PID. **Do not "simplify" this back to plain `ShellExecuteW`** ŌĆö that call returns no handle/PID at all, which used to mean HWiNFO64's dialog automation (`click_sequence`), post-launch window positioning, and PID-based `taskkill` on stop *all* silently no-op'd for it specifically (field-diagnosed via the debug log: every other tool logged `_auto_click_sequence indul`, HWiNFO64 never did). If `ShellExecuteExW` itself returns no handle (user dismissed the UAC prompt, or AppCompat is intercepting), the function still returns `-1` and automation is skipped for that run only ŌĆö this is the one remaining case `STRESS_KILL_IMAGES` name-based sweep exists for.
- **Stress-test tools ŌĆö unattended input automation**: each tool's startup dialogs are fed automatically, using a different Win32 technique per tool discovered through real hardware testing (do not regress these ŌĆö see git history for the failed intermediate attempts):
  - *Linpack* (`LINPACK_PROMPT_SCRIPT` + `_build_linpack_console_script` + `_auto_answer_console`) is driven via **`SendInput`-simulated real keystrokes** into its console window ŌĆö NOT `stdin=subprocess.PIPE` (piping stdin while also passing `CREATE_NEW_CONSOLE` made Linpack fail to start at all). The automation is **prompt-verified**: before every answer, `_read_console_screen()` (AttachConsole + `CONOUT$` + `ReadConsoleOutputCharacterW`) polls the console's visible text until the expected prompt substring actually appears (60s/step) ŌĆö never type blind on a fixed timer, on a loaded machine the menus appear seconds late and the answers land on the wrong prompt. `Linpack.exe` is a Bat2Exe-wrapped `.bat` (extractable from `%TEMP%\<rnd>.tmp` while running) whose menus read with `choice` (ONE key, no Enter) but whose run-count reads with `set /p` (line + Enter) ŌĆö hence the per-step needs-Enter flag in `LINPACK_PROMPT_SCRIPT`: sending Enter after a `choice` menu leaves the CR in the console input buffer, the next `set /p` reads it as an empty line, `RUNS` stays undefined, and the batch's `if %RUNS% LSS 1` becomes a syntax error that **silently kills the whole script** (~1s after the RAM answer ŌĆö this was the long-mysterious field crash, root-caused and fixed with a live repro). The v1.1.8 stress flow is: action menu `2` ŌåÆ RAM option ŌåÆ run count `10000` ŌåÆ threads `Y` ŌåÆ sleep `N` (the app manages power itself) ŌåÆ HWMonitor `N` (HWiNFO already runs) ŌåÆ Enter on `pause`. The RAM option comes from `_pick_linpack_ram_option()` against `LINPACK_RAM_OPTIONS` (menu-number ŌåÆ GB pairs), picking the largest option that fits the machine's *total* RAM (`_get_ram_gb()`, `GlobalMemoryStatusEx` via ctypes, total rounded *up* since an "8GB" stick reports ~7.85GB) **and** the currently *available* RAM minus ~1.5GB headroom ŌĆö on a 12GB machine running FurMark+Prime95 alongside, the total-only pick (10GB) would page-thrash the whole box. Before each simulated line, code calls `SetForegroundWindow` and then re-reads `GetForegroundWindow()` to confirm the switch actually took effect ŌĆö Windows silently refuses foreground-stealing from background processes under several conditions, so this is checked explicitly rather than assumed.
  - *FurMark / Prime95 / HWiNFO64* (`STRESS_CLICK_SEQUENCES`, `_auto_click_sequence`) get their native Win32 buttons/radio-buttons pressed via a **`BM_CLICK` message sent with `PostMessageW`** ŌĆö not coordinate-based mouse-click simulation (immune to window size/position/DPI, no foreground focus needed), and deliberately NOT `SendMessageW`: SendMessage blocks until the button's click handler returns, and FurMark's "GPU stress test" button opens a *modal* CAUTION dialog from its handler, so a SendMessage click never returns until that dialog closes ŌĆö the "GO" step could never run (field log showed the call stuck for 50s until the user manually closed the window). A sequence step is one label, a list of localized alternatives (HWiNFO's start button varies with OS display language ŌĆö `['Ind├Łt├Īs', 'Start']` ŌĆö while FurMark/Prime95 stay English regardless of locale), or a dict with `labels` plus options: `skip_if_found` (skip the step without clicking when a *later* step's control is found instead ŌĆö Prime95's GIMPS welcome dialog only appears on the *very first* launch; once clicked, `prime.txt` gets `StressTester=1` and every later launch opens straight into "Run a Torture Test"), `optional` + `timeout` (the step's dialog may legitimately never appear ŌĆö HWiNFO pops one more window *after* Start: field-identified as the "HWiNFO┬« 64 Update" notifier whose dismiss button is `Bez├Īr├Īs`/`Close`, so an optional exact-match step over `OK`/`Igen`/`Yes`/`Bez├Īr├Īs`/`Close` waits 20s and silently moves on if nothing comes; the written `HWiNFO64.INI` also sets `CheckForUpdate=0` to suppress that dialog at the source, the click step being the safety net), and `exact` (full-text match instead of substring ŌĆö mandatory for short labels like `OK`/`Igen`, where substring matching would false-positive on any Hungarian `...ok`-suffixed text in HWiNFO's sensor list). Label matching goes through `_normalize_ctrl_text()` which strips the Win32 accelerator `&` **and collapses whitespace runs** ŌĆö real control texts include `Small FFTs (tests L1/L2/L&3 caches, ...)` and `Just  &Stress Testing` (double space!), so a plain substring match can never hit either (each of these alone broke Prime95 automation in the field). Each step is located with `_find_pid_window_with_child_text()` (up to 60s poll), not `_find_window_for_pid()` (which always returns the *largest* visible window for a PID) ŌĆö a confirmation dialog is normally smaller than the app's main window, so "largest window" logic would miss it entirely. The *last* step of every sequence is additionally verified by `_verify_final_click()`: the final button (HWiNFO "Ind├Łt├Īs", Prime95 "OK", FurMark "GO") always dismisses its dialog, so the button must disappear within a few seconds ŌĆö if it doesn't, the posted click likely got lost (seen in the field on a machine already running the other 3 stress engines at 100%) and it retries, first with another `PostMessageW` and then with `SendMessageTimeoutW` (the old synchronous delivery, but with a bounded 3s wait so a modal-opening button still can't deadlock it). Mid-sequence steps don't need this ŌĆö the next step's window hunt is their implicit verification.
  - *Any console app* (`_find_console_window_for_pid`, ~line 1115, using `AttachConsole`/`GetConsoleWindow`/`FreeConsole`): a console window's true owner is usually a hidden `conhost.exe` process, not the spawned app's own PID, so plain `EnumWindows` + `GetWindowThreadProcessId` (what `_find_window_for_pid` uses) never finds it ŌĆö confirmed on real hardware (Linpack's window was still "not found" after a 30s wait while FurMark/Prime95/HWiNFO64's GUI windows were all found fine by the PID-based approach). `AttachConsole`/`FreeConsole` mutate process-global state (not per-thread), so every call is serialized through `self._console_attach_lock`.
  - Poll timeouts here are deliberately long (60s) and `start_stress_tests()` launches the 4 tools with a **3-second stagger**, not all at once ŌĆö launching FurMark+Prime95+Linpack+HWiNFO64 simultaneously (especially alongside an in-flight `dism /Get-Drivers` call, which also fires on startup for the driver-list refresh) can overload a weak machine badly enough that a confirmation dialog takes *closer to a minute* than a few seconds to appear; a real capture showed FurMark's "GO!" button still succeeding at the 56-second mark, well past what used to be a 15s timeout. If you touch these numbers, err long, not short.
  - The three long-standing field failures were root-caused and fixed with live on-hardware repros (2026-07): Prime95 = missing-welcome-dialog skip + `&`-accelerator matching, FurMark = SendMessage modal deadlock, Linpack = wrong answer list for v1.1.8 + stray Enter aborting the batch (details in the bullets above). Each fix was verified by actually launching the tool and confirming the load runs (Prime95 CPU-time delta, FurMark render window with FPS title, Linpack solver output). If a new field report comes in, start from the diagnostics bullet below ŌĆö and still prefer adding more logging over guessing at another timing fix.
- **Stress-test tools ŌĆö diagnostics**: because Win32 UI-automation failures are otherwise silent, every call in the automation path (`SendInput`, `SetForegroundWindow`, `AttachConsole`, `PostMessageW`, ...) has its return value checked and logged rather than just invoked ŌĆö e.g. `_send_unicode_char`/`_send_vk` (~line 1083/1100) return whether `SendInput` actually injected the event (0 means blocked) instead of assuming success. `_debug_dump_pid_windows(pid, context)` (~line 1040) is the go-to failure-diagnosis helper: on any "not found" path it enumerates *every* top-level window for that PID (title/class/visibility/rect) plus every child control (title/class) and dumps it as one `[STRESSTOOLS-DEBUG]`-tagged warning, so a failed match shows what *was* actually there instead of just "not found". Automation background threads are wrapped through `_run_automation_safely()` (inside `_launch_stress_exe`, ~line 1558) so an uncaught exception inside a daemon thread logs a full traceback instead of vanishing silently. When debugging a report from the field, grep the user's `DriverVar├Īzsl├│_debug.log` for `STRESSTOOLS-DEBUG` first.
- **Stress-test tools ŌĆö post-launch window layout** (`_position_stress_windows`): runs only after **all automation background threads have finished** (each `_launch_stress_exe` call appends its automation thread into the caller's `thread_sink` list; `start_stress_tests` waits on them with a 240s ceiling, then a further ~3s settle for the final windows ŌĆö e.g. FurMark's render window appears a few seconds *after* GO is clicked; the settle can be short because each automation thread already waits for its own last click to take effect via `_verify_final_click`). It used to be a fixed 30s timer, which routinely fired mid-automation: `_minimize_other_windows` minimized the still-unanswered dialogs and FurMark's render window didn't exist yet when positioning ran. Layout is 4 screen quadrants of the work area: FurMark top-left, Prime95 top-right, Linpack bottom-left, HWiNFO bottom-right. The 3 non-FurMark windows are resized to their quadrant; FurMark is *moved* to the top-left corner at native size (`SWP_NOSIZE` ŌĆö moving doesn't crop the render surface, resizing does: a forced resize just crops the picture, hiding the FPS counter) and pushed to `HWND_BOTTOM`, so exactly its top-left quarter shows in the free quadrant. `_minimize_other_windows()` then minimizes every other visible window ŌĆö including DriverVar├Īzsl├│'s own window and any leftover per-tool config dialogs ŌĆö except the 4 just-positioned ones, per explicit user request; expect the app's own progress-modal to vanish into the taskbar along with everything else once this step runs.
- **System report generation** (`generate_system_report(self, note=None)`, in `DriverToolApi`, ~line 5136) builds an HTML report via raw f-string concatenation (not a template engine), called synchronously (not via `_safe_thread`). Any value sourced from WMI/CIM or `smartctl` JSON must go through the local `g(dict, key, default)` helper (handles WMI nulls) and `e(value)` helper (HTML-escapes) before interpolation ŌĆö both are defined inline at the top of the function, not module-level. **GPU VRAM must not be read as plain `Win32_VideoController.AdapterRAM`** ŌĆö that WMI property is a 32-bit `UInt32`, and modern drivers (NVIDIA especially) return a `0xFFFFFFFF`-ish overflow sentinel for it once actual VRAM exceeds ~4GB, which reads back as almost exactly 4.0GB regardless of the card's real size (field-confirmed: a 12GB RTX 3060 reported 4.0GB). The report's PowerShell block instead cross-references each adapter's `PNPDeviceID` against the registry ŌĆö `HKLM:\SYSTEM\CurrentControlSet\Control\Class\{4d36e968-e325-11ce-bfc1-08002be10318}\00XX\MatchingDeviceId` (prefix match) ŌĆö and reads the 64-bit `HardwareInformation.qwMemorySize` value from the matching subkey (same technique GPU-Z/HWiNFO use), falling back to the raw `AdapterRAM` only if no registry match is found. The report's own `<style>` block sets `-webkit-print-color-adjust: exact` / `print-color-adjust: exact` on `*` ŌĆö without it, browsers drop background colors by default when printing/exporting to PDF (ink-saving default), which silently turns the dark "Gyors ├¢sszefoglal├│" summary box and every colored badge white in the printed/PDF output even though they render correctly on screen. The optional `note` param (passed from the UI's pre-generation prompt) renders into a `.note-section` at the end of the page as one `<div class="note-line">` per line (padded to at least 4 lines, plus one trailing blank), each with its own `border-bottom` ŌĆö **do not go back to a single `repeating-linear-gradient` background for the ruled lines**: at fractional print-DPI scaling a gradient's repeating bands rasterize with visibly inconsistent thickness line-to-line (field-confirmed), whereas one border-bottom per line div renders uniformly regardless of print scale.

## Git / release rules (IMPORTANT)

- **Never commit, push, or release on your own initiative** ŌĆö not even after finishing a fix the user asked for. Fixing code and shipping it are two separate decisions; leave changes in the working tree and tell the user they're ready, then wait for an explicit "commitold/pushold/adj ki buildet" style instruction. The bat ends with `pause`, which hangs a non-interactive shell ŌĆö run it as `cmd /c rebuild_verzioszam_novelessel_es_github_pushal.bat < nul` so the pause is auto-fed.

## Build / release

There is no test suite and no `requirements.txt`; the only third-party runtime dependency is `pywebview` (imported with a try/except guard at the top of `driver_tool.py` that exits with a clear message if missing). The build machine runs Python 3.12 + PyInstaller 6.17 (the README's "Python 3.14" mention is aspirational, not the actual toolchain).

- **Full release** (build-number bump, PyInstaller build, commit, push): run `rebuild_verzioszam_novelessel_es_github_pushal.bat` from the repo root (there is NO file named `rebuild.bat` ŌĆö older notes referred to it by that shorthand). It calls `bump_build.py` first, which increments `BUILD_NUMBER` from `max(local, GitHub-published)` and keeps `version_info.txt`'s `filevers`/`prodvers`/`FileVersion`/`ProductVersion` in sync with the new build number ŌĆö never edit `BUILD_NUMBER` by hand and commit a lower value, since that breaks the auto-updater for users already past it.
- **Manual build only** (no version bump, no commit/push): run `build.bat`, or directly `python -m PyInstaller --clean DriverVarazslo.spec`. Output is `dist/DriverVarazslo.exe`. The spec embeds `icon_drivervarazslo.ico` and `version_info.txt` (Windows file-version resource ŌĆö keep this populated and in sync with `BUILD_NUMBER`; an unsigned exe with no/stale version info is far more likely to be flagged/deleted by Defender) and explicitly disables UPX compression for the same Defender-false-positive reason. `build.bat` needs `@echo off` at the top (and `chcp 65001` for the accented console output) ŌĆö without it cmd echoes every line of the script (including each line of the `if %ERRORLEVEL%` block) before running it, which looks like the build is broken even when it isn't.
- **Run from source**: `python driver_tool.py` (GUI, needs admin ŌĆö it self-elevates via UAC) or `python driver_tool.py --cli` (text menu).
- `DriverVarazslo.spec`, `version_info.txt`, and `dist/DriverVarazslo.exe` are intentionally tracked in git (not gitignored) ŌĆö the spec and version-info files are static build inputs the release bat depends on existing, and the dist exe is what the in-app updater downloads.
- Debug logging goes to `DriverVar├Īzsl├│_debug.log` in `_app_data_dir()` (`<SystemDrive>\DriverVarazslo`, e.g. `C:\DriverVarazslo` ŌĆö NOT next to the exe, since the exe may run from a read-only location like a network share or USB stick), via a `RotatingFileHandler` (5 MB ├Ś 2 backups) ŌĆö don't swap this back to plain `logging.basicConfig(filename=...)`, the DEBUG-level log (every subprocess's stdout, up to 4000 chars each) grows unbounded otherwise, especially on a long-lived service USB/WinPE install run across many machines. `generate_system_report`'s HTML output goes to the same `_app_data_dir()` folder, for the same reason (previously both were written next to the exe).
