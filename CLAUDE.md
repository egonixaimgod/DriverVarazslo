# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

DriverVarázsló is a Windows driver-management utility (Hungarian UI/comments) distributed as a single PyInstaller-built `.exe`. It manages third-party/inbox drivers (list, force-delete, backup/restore, offline WIM/ESD extraction), can repair non-booting Windows installs (BCD/bootloader rebuild from WinPE), pauses Windows Update driver pushes, removes ghost devices, runs a one-click "AutoFix" routine that chains a reboot via a scheduled task, generates an HTML hardware report with S.M.A.R.T. data, and launches bundled stability-test tools (FurMark/Prime95/Linpack/HWiNFO/HD Sentinel) with screen-sleep temporarily disabled and their startup dialogs auto-answered. See [README.md](README.md) for the full feature list.

## Architecture

- **`driver_tool.py`** — the entire application (~6550 lines), two API classes:
  - `DriverToolApi` (line ~277) — the GUI backend, exposed to the frontend as `pywebview`'s `js_api`. Nearly every feature is a method on this class.
  - `CliApi` (line ~4694) — a parallel, full-featured mirror of `DriverToolApi` for the text-menu CLI fallback (`run_cli_mode()` ~line 5989, `main_menu()` ~line 6008). When adding a feature to one, the other usually needs the equivalent — this drifts easily (e.g. CliApi previously lacked WU pause/resume and standalone ghost-device deletion, had a materially weaker `autofix()` than the GUI's `run_autofix`, and its offline DISM `/Add-Driver` call was missing `/ForceUnsigned` that the GUI had; all fixed, but this class of gap can reappear — note that CliApi's `autofix()` is *intentionally* a simpler one-shot design with no reboot-chaining, documented in its own printed output, not a bug to "fix" into parity). The Stress-Test feature (below) is the one deliberate, permanent exception: it's GUI-only — CliApi's own help menu just tells the user to relaunch without `--cli` for it, since it depends on `pywebview`'s window plus Win32 UI-automation that has no meaningful CLI equivalent.
  - Section banners (`# ====...`) mark major feature groups within `DriverToolApi`: driver listing, BCD repair, driver deletion, hardware/ghost-device scan, WU driver install, WU pause/management, backup/restore. `grep -n "# ===="` to navigate.
- **`ui.html`** — the entire frontend: plain HTML/CSS/JS (no framework, no build step), dark "glassmorphism" theme. Talks to Python only through `window.pywebview.api.<method>()` (helper: `api()`) and receives async push events via `window.handlePyEvent(payload)`, which Python calls from `DriverToolApi.emit(event, data)` (`driver_tool.py` ~line 337) via `window.evaluate_js(...)`. JS console errors/uncaught exceptions are forwarded back to the Python logger through `api().js_log(...)`.
- **GUI vs CLI dispatch**: `if __name__ == "__main__"` (driver_tool.py ~line 6235) does a single-instance mutex check, then UAC self-elevates if not admin (both the `--cli` and GUI paths call the shared `_relaunch_elevated()` helper, ~line 6257), then branches: `--cli` arg or `check_webview2_runtime()` failing (WebView2 missing/too old, min v109) routes to `run_cli_mode()`; otherwise it creates the `pywebview` window with `DriverToolApi` as `js_api`.
- **Offline mode**: most driver operations check `self.target_os_path` — when set (via `change_target_os`/`apply_target_os`), operations target a foreign Windows volume (e.g. `D:\Windows`, mounted from WinPE or a second OS) using `dism /Image:...` instead of the live `/Online` system. Always handle both branches when touching driver listing/deletion code. DISM calls that parse English text output (driver listing) must pass `/English` on **both** the GUI and CLI paths, or output on a non-English Windows/WinPE silently fails to parse.
- **WinPE awareness**: code checks `os.environ.get('SystemDrive') == 'X:'` to detect running from WinPE/boot media and redirects large temp downloads/extracts to `C:\DV_Temp` instead of the RAM-disk `X:` drive — this redirect must not depend on any other condition (e.g. whether an offline target OS is also selected), since the whole point is that `X:` is a tiny RAM disk.
- **Packaging**: `resource_path()` (~line 164) resolves bundled resources (`ui.html`) via `sys._MEIPASS` when frozen. `BUILD_NUMBER` (top of file) drives the in-app auto-updater (`check_for_updates`, ~line 450): it fetches `driver_tool.py` from `raw.githubusercontent.com`, compares `BUILD_NUMBER`, and if newer, downloads and replaces `dist/DriverVarazslo.exe` — this published exe is intentionally committed to the repo (`git add -f`) so the updater has something to fetch. Never let an automated/manual edit lower the committed `BUILD_NUMBER` — `check_for_updates` only offers an update on strict `new_build > BUILD_NUMBER`, so a regression silently freezes already-updated users (this happened once; see `bump_build.py`). All network downloads (`check_for_updates`, `perform_update`, `_download_stresstools`, `bump_build.py`) use `ssl.create_default_context()` with certificate/hostname verification **on** — never add `check_hostname = False` / `verify_mode = ssl.CERT_NONE` here, these paths download an `.exe` (or other update payload) that gets executed with admin rights.
- **AutoFix resume-after-reboot**: the AutoFix flow (`run_autofix`, ~line 2851) writes a scheduled task pointing back at the exe with `--resume-autofix` / `--resume-step1`, since cleanup steps (Fast Startup off, ghost device removal) require a reboot mid-flow before driver reinstallation can continue. It's a 3-leg chain: Leg A (fresh start) shows the confirm dialog, pauses WU, schedules a task with `--resume-step1`, reboots. Leg B (relaunched by that task — a brand new process) is where ghost-device/driver deletion actually happens, then it schedules `--resume-autofix` and reboots again. Leg C (`--resume-autofix`) does the WU rescan/install, looping (reboot + re-resume) while new drivers keep getting installed. **Every reboot in this chain uses `shutdown /r /t 0 /f`** (the `/f` force-closes blocking apps) — a reboot without `/f` can hang the unattended chain indefinitely. **Any user choice made in the confirm dialog (Leg A) that needs to affect Leg B's behavior cannot be passed as a plain function parameter** — Leg B is a different OS process with no memory of Leg A's call. Thread it through as an extra flag appended to Leg A's scheduled-task `-Argument` string (alongside `--resume-step1`), then parse it back out of `sys.argv` in `__init__` (same pattern as `self.resume_step1`/`self.resume_mode`) so Leg B can read it via `self.<flag>`. See `skip_printer_drivers`/`AUTOFIX_PRINTER_SKIP_CLASSES` for a worked example. The scheduled-task PowerShell command is built via an f-string with single-quoted `-Execute`/`-Argument` values — always pass the exe path/args through the `_ps_quote()` helper (escapes `'` as `''`) before interpolating, or a path containing an apostrophe (e.g. a username) breaks the resume chain.
- **Subprocess-heavy backend**: most real work shells out via `self._run(cmd, **kwargs)` (a logging wrapper around `subprocess.run`, two near-identical copies exist — one per API class) to `dism`, `pnputil`, `bcdboot`/`bootrec`, `takeown`/`icacls`, `powercfg`, and inline PowerShell (`-Command` heredoc strings) for WMI/CIM queries and Windows Update COM API access.
- **Background task concurrency (`DriverToolApi` only)**: long-running tasks run via `self._safe_thread(task, target)` (~line 401), which centralizes two things: (1) a single `self._task_busy` gate — if any `_safe_thread` task is already running, a new one is rejected with a toast instead of starting, because two concurrent tasks would otherwise stomp on shared state (`self._cancel_flag`, `self.hw_updates_pool`); (2) resetting `self._cancel_flag = False` for the new task, done *only after* the busy-check passes (never reset it in the calling method itself, or a rejected/queued call would silently clear a cancel-in-progress on the task that's actually running). `start_hw_scan()` doesn't use `_safe_thread` (it has its own `_hw_scanning` flag plus emits `hw_scan_result` on every exit path so the UI progress bar never gets stuck) but still participates in the same `self._task_busy` gate for the same reason — installing drivers while a hardware scan is rewriting `self.hw_updates_pool` is a data race. Individual stress-tool launches (`start_stress_tool`, below) are the deliberate exception: they bypass `_task_busy` entirely so the user can fire off several tools back-to-back, relying instead on a narrower `self._stresstools_download_lock` around just the shared-ZIP download/extract step. `load_drivers()` also bypasses `_task_busy` (raw `threading.Thread`) since it's a harmless read-only refresh.
- **Stress-test tools — download & power lock** (`STRESS_TOOLS` dict ~line 185, `STRESS_TOOLS_BULK` ~line 199): FurMark/Prime95/Linpack/HWiNFO64/HD Sentinel ship in one externally-hosted `stresstools.zip` (GitHub release asset, not in this repo — whether a given tool actually launches depends on that ZIP's contents, e.g. an HD Sentinel executable must exist in it). `_download_stresstools()` downloads+extracts it once (marker file gates re-download, `self._stresstools_download_lock` serializes concurrent callers so the "start all" button and an individual per-tool click can't corrupt each other's extraction). `start_stress_tests()` (~line 1215, "start all") only launches `STRESS_TOOLS_BULK` — HD Sentinel is excluded on purpose (it's a disk-health *monitor*, not a load generator) and only reachable via `start_stress_tool('hdsentinel')`. Individual per-tool launches (`start_stress_tool`, ~line 1298) deliberately skip the `task_start`/progress-modal system, using only lightweight `toast` events (explicit "no popup, just launch" product requirement). Both call `_lock_power_for_stress()` (~line 615) first, which disables monitor/standby/hibernate timeouts (AC+DC, via `powercfg /setacvalueindex`/`/setdcvalueindex` on `SUB_VIDEO`/`SUB_SLEEP`); the *original* values are queried once (`_query_power_setting`, parses `powercfg /query` output *positionally* by `0x...` hex index rather than by localized label text — the labels are localized, the hex values and their AC-then-DC order are not) and saved to `HKLM\<STRESS_POWER_REG_KEY>` the first time only, then restored and cleared on the *next app startup* (`_restore_power_after_stress()`, called from `__init__` ~line 646) — there is no "test finished" signal to restore on, since the launched tools are independent detached processes.
- **Stress-test tools — unattended input automation**: each tool's startup dialogs are fed automatically, using a different Win32 technique per tool discovered through real hardware testing (do not regress these — see git history for the failed intermediate attempts):
  - *Linpack* (`_build_linpack_stdin` + `_auto_answer_console` ~line 803) needs its console menu (pick a RAM size, then three Y/N prompts) driven via **`SendInput`-simulated real keystrokes** into its console window — NOT `stdin=subprocess.PIPE`. Piping stdin while also passing `CREATE_NEW_CONSOLE` made Linpack fail to start at all (its console-mode input handling doesn't tolerate a redirected pipe). The RAM-size menu choice comes from `_pick_linpack_ram_option()` against `LINPACK_RAM_OPTIONS` (~line 204: menu-number → GB pairs), picking the largest option that still fits the machine's actual RAM from `_get_total_ram_gb()` (`GlobalMemoryStatusEx` via ctypes, ~line 709) — rounded *up*, since Windows always reports a bit less than the marketed capacity (an "8GB" stick reports ~7.85GB).
  - *FurMark / Prime95 / HWiNFO64* (`STRESS_CLICK_SEQUENCES` ~line 209, `_auto_click_sequence` ~line 914) get their native Win32 buttons/radio-buttons pressed via a **`BM_CLICK` message** (`SendMessageW`) — not coordinate-based mouse-click simulation, so it's immune to window size/position/DPI. A sequence step is either one label or a list of localized alternatives (HWiNFO's start button varies with OS display language — `['Indítás', 'Start']` — while FurMark/Prime95 stay English regardless of locale). Each step is located with `_find_pid_window_with_child_text()` (~line 872, up to 60s poll), not `_find_window_for_pid()` (~line 1020, which always returns the *largest* visible window for a PID) — a confirmation dialog is normally smaller than the app's main window, so "largest window" logic would miss it entirely.
  - *Any console app* (`_find_console_window_for_pid`, ~line 767, using `AttachConsole`/`GetConsoleWindow`/`FreeConsole`): a console window's true owner is usually a hidden `conhost.exe` process, not the spawned app's own PID, so plain `EnumWindows` + `GetWindowThreadProcessId` (what `_find_window_for_pid` uses) never finds it — confirmed on real hardware (Linpack's window was still "not found" after a 30s wait while FurMark/Prime95/HWiNFO64's GUI windows were all found fine by the PID-based approach). `AttachConsole`/`FreeConsole` mutate process-global state (not per-thread), so every call is serialized through `self._console_attach_lock`.
  - Poll timeouts here are deliberately long (60s) and `start_stress_tests()` launches the 4 tools with a **3-second stagger**, not all at once — launching FurMark+Prime95+Linpack+HWiNFO64 simultaneously (especially alongside an in-flight `dism /Get-Drivers` call, which also fires on startup for the driver-list refresh) can overload a weak machine badly enough that a confirmation dialog takes *closer to a minute* than a few seconds to appear; a real capture showed FurMark's "GO!" button still succeeding at the 56-second mark, well past what used to be a 15s timeout. If you touch these numbers, err long, not short.
- **Stress-test tools — post-launch window layout** (`_position_stress_windows`, ~line 1051, runs ~30s after `start_stress_tests` finishes launching everything): tiles Prime95/Linpack/HWiNFO64 into 3 screen quadrants via `SetWindowPos`, but deliberately leaves FurMark's window size/position untouched and only pushes it to `HWND_BOTTOM` in z-order — force-resizing FurMark's window doesn't rescale its render surface, it just crops the picture (hiding the FPS counter in the process), so it's better left full-size but behind the other 3. `_minimize_other_windows()` (~line 1123) then minimizes every other visible window — including DriverVarázsló's own window and any leftover per-tool config dialogs — except the 4 just-positioned ones, per explicit user request; expect the app's own progress-modal to vanish into the taskbar along with everything else once this step runs.
- **System report generation** (`generate_system_report`, in `DriverToolApi`, ~line 4271) builds an HTML report via raw f-string concatenation (not a template engine), called synchronously (not via `_safe_thread`). Any value sourced from WMI/CIM or `smartctl` JSON must go through the local `g(dict, key, default)` helper (handles WMI nulls) and `e(value)` helper (HTML-escapes) before interpolation — both are defined inline at the top of the function, not module-level.

## Build / release

There is no test suite and no `requirements.txt`; the only third-party runtime dependency is `pywebview` (imported with a try/except guard at the top of `driver_tool.py` that exits with a clear message if missing).

- **Full release** (build-number bump, PyInstaller build, commit, push): run `rebuild.bat` from the repo root. It calls `bump_build.py` first, which increments `BUILD_NUMBER` from `max(local, GitHub-published)` and keeps `version_info.txt`'s `filevers`/`prodvers`/`FileVersion`/`ProductVersion` in sync with the new build number — never edit `BUILD_NUMBER` by hand and commit a lower value, since that breaks the auto-updater for users already past it.
- **Manual build only** (no commit/push): `python -m PyInstaller --clean DriverVarazslo.spec`. Output is `dist/DriverVarazslo.exe`. The spec embeds `icon_drivervarazslo.ico` and `version_info.txt` (Windows file-version resource — keep this populated and in sync with `BUILD_NUMBER`; an unsigned exe with no/stale version info is far more likely to be flagged/deleted by Defender) and explicitly disables UPX compression for the same Defender-false-positive reason.
- **Run from source**: `python driver_tool.py` (GUI, needs admin — it self-elevates via UAC) or `python driver_tool.py --cli` (text menu).
- `DriverVarazslo.spec`, `version_info.txt`, and `dist/DriverVarazslo.exe` are intentionally tracked in git (not gitignored) — the spec and version-info files are static build inputs `rebuild.bat` depends on existing, and the dist exe is what the in-app updater downloads.
- Debug logging goes to `DriverVarázsló_debug.log` next to the exe (or script when run from source), via a `RotatingFileHandler` (5 MB × 2 backups) — don't swap this back to plain `logging.basicConfig(filename=...)`, the DEBUG-level log (every subprocess's stdout, up to 4000 chars each) grows unbounded otherwise, especially on a long-lived service USB/WinPE install run across many machines.
