## Project overview

DriverVarázsló is a Windows driver-management utility (Hungarian UI/comments) distributed as a single PyInstaller-built `.exe`. It manages third-party/inbox drivers (list, force-delete, backup/restore, offline WIM/ESD extraction), can repair non-booting Windows installs (BCD/bootloader rebuild from WinPE), pauses Windows Update driver pushes, removes ghost devices, runs a one-click "AutoFix" routine that chains a reboot via a scheduled task, generates an HTML hardware report with S.M.A.R.T. data (one-click printable to the shop's network printer), launches bundled stability-test tools (FurMark/Prime95/Linpack/HWiNFO/HD Sentinel) with screen-sleep temporarily disabled and their startup dialogs auto-answered, shows/disables BitLocker on the system drive, and downloads a standalone firewall net-blocker batch script. See [README.md](README.md) for the full feature list — but note the README has drifted from the code: it describes the AutoFix reboot chain as using the `RunOnce` registry key (it is a **scheduled task**, `DriverVarazsloResume` — there is no `RunOnce` anywhere in the tree), names the release script `rebuild.bat` (the real name is `rebuild_verzioszam_novelessel_es_github_pushal.bat`), claims an `atexit` handler kills orphan processes (see the process-exit rules below — `atexit` would never fire), and says Python 3.14 (the build machine runs 3.12). Trust the code and this file over the README.

## Commands

**Do NOT build the exe yourself — the user builds it when they want to** (explicit user decision). They have the two build scripts in the project folder (`build.bat`, `rebuild_verzioszam_novelessel_es_github_pushal.bat`) and run them by hand. Make and verify your code changes (syntax check + importing the API classes + driving the pure/mixin logic against live data — see below), then tell the user it's ready; leave the actual PyInstaller build/release to them unless they explicitly ask you to build. This is the build-side twin of the "never commit/push on your own initiative" rule.

There is **no test suite, no linter config, and no `requirements.txt`** in this repo — "verifying a change" ultimately means building the exe and running it, but that build is the *user's* step, not yours.

```bash
python driver_tool.py            # run from source (GUI; self-elevates via UAC)
python driver_tool.py --cli      # run from source, force the text-menu CLI
python -m PyInstaller --clean --noconfirm DriverVarazslo.spec   # build -> dist/DriverVarazslo.exe
python -m py_compile app/gui/autofix.py             # cheapest syntax check while editing
python -c "import app.gui.api, app.cli.api"         # both API classes still assemble
cmd /c rebuild_verzioszam_novelessel_es_github_pushal.bat < nul   # FULL RELEASE - only on explicit request
```

**Both `.bat` files end with `pause`.** Call PyInstaller directly (as above) rather than `build.bat`, or the build appears to produce no output at all and hangs waiting for a keypress. Same reason `< nul` is mandatory on the release script — which bumps `BUILD_NUMBER`, builds, commits and **pushes**; see the Git / release rules before ever running it.

### Verifying logic without a machine to break

"No test suite" does **not** mean "nothing is testable". The `app/*_core.py` modules import nothing from `ctypes`/`pywebview` and never call a subprocess themselves — **every function that needs one takes `run` as a parameter** — and the parsing/decision helpers are pure functions. So the risky logic — HWID matching, DISM output parsing, downgrade/duplicate filters, delete-result classification — can be exercised in-process, in a second, with a stub:

```python
from app import drivers_core
drivers_core.parse_dism_driver_list(open('sample_dism_output.txt').read())   # pure

class R: returncode, stdout, stderr = 0, 'Driver package deleted successfully.', ''
calls = []
def fake_run(cmd, **kw): calls.append(cmd); return R()
drivers_core.delete_driver_package(fake_run, 'oem7.inf')   # asserts on `calls`
```

Do this for any change to matching/parsing/filtering rules, using **hardware IDs and command output copied verbatim from the field log** — that is how the R9-200 GPU-matching bug was pinned down and confirmed fixed. Put throwaway scripts in the scratchpad, not the repo.

**The GUI mixins are drivable too — and on a Windows dev box they can run against *live* data, which is stronger than any captured sample.** A mixin only needs `_run` and `emit`; nothing forces you to build the whole `DriverToolApi`:

```python
class F(GuiHwScanMixin):
    def _run(self, cmd, **kw): return subprocess.run(cmd, capture_output=True, text=True, encoding='utf-8')
    def emit(self, *a, **k): pass
f = F()
f._catalog_search_collect(candidates, f._get_installed_driver_info())   # real WMI + real catalog
```

This is how the "vendor driver instead of the generic one" work was validated end-to-end without ever installing anything: real `Win32_PnPEntity`/`Win32_PnPSignedDriver` output, real `catalog.update.microsoft.com` queries, decisions printed instead of executed. It is also what exposed that the audio device advertises **only** SUBSYS-scoped hardware IDs — a fact no captured sample would have suggested. Pair it with a stubbed `_catalog_fetch_rows` (returning hand-written rows) to unit-test the *decision* logic offline: which row wins, when a version gate applies, when nothing is offered.

Still out of reach without a real machine and `dist/DriverVarazslo.exe`: anything that changes the device stack (installs/deletes), the stress automation, and the AutoFix reboot chain.

### Field logs are the primary bug report

Bugs in this project arrive as a pasted `DriverVarázsló_debug.log` from a shop machine, not as a stack trace. Reading one:

- Pair a `[CMD]` line with its result by the **`[threadName]`** — parallel workers interleave, and a command's output can otherwise look like it belongs to a different call.
- `[CALL] Class.method(...)` / `-> return (elapsed)` brackets every API method; a `[CALL]` with no matching `->` is where execution stopped.
- Grep `STRESSTOOLS-DEBUG` for stress-test reports, `[EMIT:` for what the user actually saw on screen, `[UI-AKCIO]`/`[CLI-INPUT]` for what they clicked or typed.
- **A log that ends mid-line followed by a run of blank space is not a truncated paste** — it means the machine went down hard (BSOD/power loss) with the file's tail still in the page cache. Work that happened after the last readable line is invisible; reconstruct it from state instead (e.g. the third-party package count in the *next* session's startup dump). This is how the broken-chain bug was found.
- The log's Hungarian text may render as `Ã¡`-style mojibake in the paste, and a few tools (`reg.exe`, `pnputil`) emit genuinely mis-encoded bytes; both are display-only, never a bug in itself.

## Code layout (refactored 2026-07 from a single 8000-line driver_tool.py)

- **`driver_tool.py`** — thin entry point ONLY: the `BUILD_NUMBER` literal + the `__main__` block (single-instance mutex, UAC self-elevation via `_relaunch_elevated()`, logging setup, GUI-vs-CLI dispatch, WebView2 install offer + watchdog). **The `BUILD_NUMBER = <int>` line must stay in this file, at column 0, forever**: every already-deployed exe's auto-updater downloads *this file* from GitHub and regexes `^BUILD_NUMBER\s*=\s*(\d+)` out of it (and `bump_build.py` edits it here) — moving it to another file would permanently freeze all fielded installs on their current build. At startup the entry point copies it into `app.common.BUILD_NUMBER` (a module attribute defaulting to 0); **all other code reads `common.BUILD_NUMBER`**, never imports it from `driver_tool` (that would re-execute the module and duplicate state).

### Code organization rules (IMPORTANT — this structure must not erode back into a monolith)

- **A new feature = a new file**, following the existing pattern: new mixin in `app/gui/<feature>.py` (and `app/cli/<feature>.py` if it has a CLI equivalent), registered in the base-class list of the matching `api.py`. Never bolt a new feature's methods onto an existing unrelated mixin, and never grow `base.py` into a dumping ground — `base.py` is only for genuinely cross-feature plumbing (`emit`, `_run`, `_safe_thread`, dialogs, target-OS state).
- **Logic shared by GUI and CLI goes in a module-level `app/<feature>_core.py`**, called by both thin mixins — never duplicated into both classes; duplication is what caused the Build-192-era "one path works, the other silently broken" bugs. Since the 2026-07 dedup pass this is fully enforced: every feature with both a GUI and CLI face has its `_core` module (see the Code layout list), the mixins only do presentation (emit vs. print) and threading/cancel wiring. Long core sequences take a `log(msg)` callback (GUI passes a `task_progress`-emitting lambda, CLI passes `print`) and an optional `check_cancel()` callback (CLI passes None).
- **Constants/ctypes**: feature-local constants live at the top of the feature's own file (see `storeprint.py`); constants shared across files go to `app/stress_defs.py`-style def modules or `app/common.py`; ctypes structs go to `app/win32.py`.
- Keep files roughly feature-sized. The biggest are `app/gui/hwscan.py`, `app/gui/autofix.py`, `app/wu_core.py` and `app/gui/stress_automation.py` (all four ~900–1200 lines and still growing); if one grows past that because it really contains two concerns, split it the way `stress.py`/`stress_automation.py` was split. Don't chase an exact line budget — the question is whether the file is still *one* feature.
- **When you add/rename/split a file, update the "Code layout" list above in the same change** — this map is what lets a fresh session navigate without grepping the whole tree.
- **`ui.html`** — the entire frontend: plain HTML/CSS/JS (no framework, no build step), dark "glassmorphism" theme **with a unified hero-card design system (2026-07 facelift)**: every view opens with a `.view-hero` header card — big title + description + `.hero-chip` feature chips on the left, the main action button(s) in `.vh-actions` on the right, and a faint emoji watermark taken from the `data-emoji` attribute. Color variants: default purple, `.vh-red` (dangerous/blocking: BitLocker, block.bat), `.vh-blue` (Windows/system: WU, HW scan), `.vh-teal` (backup/report). Two bespoke heroes predate the generic class and stay as-is: `.stress-hero` (orange, Stress Teszt) and `.autofix-hero` (purple, the AutoFix quick-launch card in the Drivers view; big button style `.btn-autofix-hero`). The Stress view additionally has the `.tool-grid`/`.tool-card` service-program card grid. **New views/features must follow this hero pattern** (title/desc/chips + action right), and existing element `id`-s must never change during styling (JS references them). Sidebar label for the stress view: "Stress Teszt & Szervíz Programok". Drivers-view order (explicit user decision): welcome hero (🧙‍♂️, generic `.view-hero` with the `.welcome-features` grid at `flex-basis:100%`) on top → third-party driver table panel → AutoFix quick-launch `.autofix-hero` at the bottom. Talks to Python only through `window.pywebview.api.<method>()` (helper: `api()`) and receives async push events via `window.handlePyEvent(payload)`, which Python calls from `DriverToolApi.emit(event, data)` (`app/gui/base.py`) via `window.evaluate_js(...)`. JS console errors/uncaught exceptions are forwarded back to the Python logger through `api().js_log(...)`. The refactor did not change the JS↔Python contract: every `api().<name>()` call still resolves to a `DriverToolApi` method (verified programmatically after the split).
- **`__file__` is a trap after the split**: any "path of the running program" logic must use `app.common._app_exe_path()` (frozen → `sys.executable`, source → repo-root `driver_tool.py`), NOT `os.path.abspath(__file__)` — inside `app/` that now points at the submodule and would break e.g. the AutoFix scheduled-task exe path when running from source. All five original occurrences were converted; don't reintroduce the raw pattern.

## Logging (deliberately maximal — the debug log is the primary field-diagnosis tool)

- Log file: `DriverVarázsló_debug.log` in `_app_data_dir()` (`<SystemDrive>\DriverVarazslo`, NOT next to the exe — it may run from a read-only USB/network share), `RotatingFileHandler` 5 MB × 2 backups. Don't swap back to plain `basicConfig(filename=...)` — the DEBUG-level log grows unbounded otherwise. Logging is configured in `driver_tool.py` **before** the `--cli` branch (it used to be after, so CLI runs logged nothing — that was fixed during the refactor on purpose; keep it that way). The handler is `_BomRotatingFileHandler` (in `driver_tool.py`): it writes a UTF-8 **BOM** at the start of every *new/empty* log file (initial creation and post-rollover) so plain Notepad reliably detects UTF-8 instead of guessing ANSI and rendering the Hungarian text as `Ã¡`-style mojibake (field-seen) — keep the BOM logic if the handler is ever touched. Note the log *content* can still contain per-tool garble for a few localized console tools (`reg.exe` emits OEM cp852, `pnputil` emits ANSI — Windows is inconsistent per tool): those lines are display-only, never parsed, and deliberately left alone rather than risking the ASCII-based parsing paths (`dism /English`, powercfg hex positions) with a global decode change.
- **Call-logging layer** (`app/common.py`: `install_call_logging`, applied to both API classes in their `api.py`): every method of `DriverToolApi` and `CliApi` — public and private, inherited included — is wrapped to log `[CALL] Class.method(args...)` on entry, `[CALL] ... -> return (elapsed)` on exit, and `[CALL] ... KIVÉTEL` with full traceback on exception (public methods at INFO, `_private` at DEBUG; arg/return reprs truncated to 200/300 chars). A small `_CALL_LOG_EXCLUDE` set skips methods that either have richer logging of their own (`emit`, `_run`, `js_log`) or run in per-keystroke/per-poll hot loops that would churn the rotating log (`_send_unicode_char`, `_read_console_screen`, ...). Static methods are skipped by design (no `self`). If you add a new hot-loop helper, add it to the exclude set; everything else gets logged automatically — no per-method logging boilerplate needed for entry/exit anymore.
- The log format includes **`[%(threadName)s]`**: parallel workers' lines interleave in the file, and without the thread name a `[CMD]` command could not be paired with its result (field-seen: a PowerShell command apparently "returning" another thread's DISM output). All background threads carry meaningful names (`task-<name>` from `_safe_thread`, plus `drivers-load`, `dup-list`, `hw-scan`, `catalog-N`, `wu-reader`, `stress-tool`/`stress-stop`, `bootfixer-dl`, `auto:<tool>`) — keep naming new threads.
- Existing layers kept: `_run` logs every subprocess command + returncode + up to 4000 chars of stdout/stderr (`[CMD]`/`[CMD_CLI]`); `emit` logs every UI push event (`[EMIT:...]`); JS errors arrive as `[JS_UI]`; stress automation diagnostics are `[STRESSTOOLS-DEBUG]` (grep that first for stress-test field reports).
- **Command timeouts + the "process won't even start" case** (both `_run`s, `app/common.py`): `_run` passes `timeout=` through to `subprocess.run` but **catches `TimeoutExpired` and returns `CommandResult(CMD_TIMEOUT_RETURNCODE, ...)` instead of raising** — a hung helper must not kill a whole AutoFix leg. Callers that care check `res.returncode == CMD_TIMEOUT_RETURNCODE`. Separately, `spawn_failed(res)` detects returncode `0xC0000142` (`STATUS_DLL_INIT_FAILED`): that is **not** "the command failed", it is "Windows could not initialize a new process" (session ending / collapsed device stack), and **every subsequent spawn will be an instant no-op**. Field-proven (Build 218): after a hung storage-driver delete, 17 further `pnputil` calls "succeeded" in 0.0s each and the leg reported `✅ Driverek eltávolítva` and rebooted, having deleted nothing. Any loop that spawns per item MUST abort on `spawn_failed` with a visible error rather than continue — a silent false success is the worst possible outcome here. `delete_driver_package` takes `timeout=` (`drivers_core.DELETE_DRIVER_TIMEOUT` = 180s) and both AutoFix delete loops honour both rules.
- **Expected non-zero exit codes**: both `_run`s take a keyword-only `ok_codes=(0,)` — codes listed there log at DEBUG as "OK - várt kód" instead of WARNING, so WARNING lines mean real anomalies. Used for: `reg delete` on a possibly-absent key (1), the idempotent final `Unregister-ScheduledTask` (1), `pnputil /delete-driver` + `wusa`/`dism /Add-Package` reboot-required (3010), `stop_stress_tests`' `taskkill` calls (128 = "no such process" — the user may have closed the tool already, or the PID is a finished Benchmark launch). Return value is unaffected — only the log level. When adding a call whose failure is a *normal* outcome, pass `ok_codes` instead of letting it cry wolf.
- **Every user action is logged**: the GUI logs every click and field change from `ui.html` (capture-phase `click`/`change` listeners → `js_log('DEBUG', '[UI-AKCIO] ...')`; text inputs only on change, never per-keystroke) so a field log replays what the user pressed and in what order; the CLI wraps `builtins.input` in `run_cli_mode()` (`_install_input_logging`, idempotent) so every menu choice/path/confirmation logs as `[CLI-INPUT] prompt -> value`.
- **No silent `except: pass`**: cleanup/cache/best-effort failures log at `logging.debug` with context; swallowed failures that *change behavior* (e.g. the AutoFix PnP JSON parse — an empty parse silently empties the device list) log at `logging.warning`. Pure value-parse fallbacks that return a semantic default (`_dup_version_key`, report `g()`/`e()`) may stay silent. Don't add new bare `except: pass` blocks.
- Both `_run` copies default `stdin` to `subprocess.DEVNULL` — do not remove: after the stress-test automation's `AttachConsole`/`FreeConsole` cycles the inherited stdin handle goes stale, and inherited-stdin + `capture_output` then fails *every* later command with `[WinError 6]` (field-proven: `taskkill` stopped working right after a stress run).

## Architecture / field-proven behavior rules

### Process model, dispatch & platform rules

- **GUI vs CLI dispatch** (`driver_tool.py` `__main__`): single-instance mutex → UAC self-elevate if not admin (both paths use the shared `_relaunch_elevated()`) → `--cli` arg or `check_webview2_runtime()` failing (min v109) routes to `run_cli_mode()` (with a WebView2 silent-install offer first in the GUI path); otherwise `pywebview` window with `DriverToolApi` as `js_api` + a 60s watchdog that falls back to CLI if `_webview_ready` never fires.
- **Process exit is always `cleanup_zombies()` + `os._exit(0)`** (`driver_tool.py`), never a plain `return`/`sys.exit()` from `__main__`: long-running helpers (`dism`, `pnputil`, PowerShell) and daemon threads can otherwise outlive the window and hold file locks / keep the single-instance mutex, and the WebView2 host thread does not always unwind. `cleanup_zombies()` taskkills our **own** process tree (`taskkill /F /T /PID <self>`). Consequence to respect: **`atexit` hooks and `finally` blocks above the exit point never run** — `os._exit` bypasses them — so cleanup that must happen (temp files, restored power settings, unregistered scheduled tasks) has to be done explicitly *before* the exit call, not registered for later. This is why `cleanup_zombies` is called manually at each exit site instead of via `atexit` (the README's claim of an `atexit` handler is wrong).
- **Unhandled exceptions are routed, not printed**: `sys.excepthook` and `threading.excepthook` log with full traceback and, if the message mentions WebView2/ICoreWebView2/.NET, set `_webview_error` so the watchdog can fall back to the CLI instead of leaving the user with a dead window.
- **GUI/CLI feature parity drifts easily**: when adding a feature to one API class, the other usually needs the equivalent (historic gaps: WU pause/resume, ghost-device deletion, `/ForceUnsigned` on offline restore — all fixed once). `CliApi.autofix()` is *intentionally* a simpler one-shot design with no reboot-chaining (documented in its printed output) — not a bug to "fix" into parity. Deliberately GUI-only (listed in the CLI's "GUI-only funkciók" menu item): Stress-Test (incl. the service-tool individual launches + Program Files install, `toolsinstall.py`), the Benchmark view (Cinebench R20 / Unigine Heaven runs + cloud leaderboard, `app/gui/benchmark.py`), the manual WU scan + selective install (hwscan), the NVIDIA/AMD/Intel/OEM vendor-driver cards, store-printer report printing, and the BootFixer.cmd download button (the CLI keeps the built-in BCD repair instead). Everything else is available in both, and since the 2026-07 dedup pass the shared body always lives in an `app/<feature>_core.py` — extend the core, never duplicate into the classes.
- **Offline mode**: most driver operations check `self.target_os_path` — when set, operations target a foreign Windows volume using `dism /Image:...` instead of `/Online`. **The running system's drive root can never become the offline target**: `apply_target_os` (`app/gui/base.py`) detects when the picked folder equals `%SystemDrive%\` and stays in live mode (returns `False`, toast explains; the JS `changeTarget` then resets the target label) — `dism /Image:C:\` against the running Windows fails with error 87 ("points to a running Windows installation") and every list silently came back as "0 driver" (field log, Build 208). Always handle both branches when touching driver listing/deletion. DISM calls that parse English text output must pass `/English`, or non-English Windows/WinPE silently fails to parse. That text parsing lives in ONE place since the dedup pass — `app/drivers_core.py: parse_dism_driver_list` (it used to exist in 4 near-identical copies) — and keys off substrings like `"Published Name"`/`"Provider Name"`/`"Class Name"`; the driver version in real `dism /Get-Drivers` output is a standalone `Version :` line (**not** `"Date and Version"`, which never matched and silently left the Version column blank until field-diagnosed).
- **`_filter_wu_scan_devices` device categorization is case-insensitive on `PNPClass`**, and each device dict carries the raw `pclass` alongside the human `cat` label. `Win32_PnPEntity.PNPClass` capitalization varies per device (audio reports `MEDIA` in caps), so the class comparisons lowercase both sides — a case-sensitive `== "Media"` dumped every audio device into "🔧 Egyéb" and, worse, would now misfire the generic-driver-replacement class gate that keys off `pclass`. Keep `pclass` on the dict and keep the comparisons lowercased.
- **`_check_internet` (`app/gui/base.py`) must not use `socket.setdefaulttimeout`**: that sets the default for the *whole process* and never resets, so after the first check every later socket that omits an explicit timeout silently gets 3s — a single slow second would abort a large download (NVIDIA driver, stresstools.zip). Use a per-call `socket.create_connection((host, port), timeout=3)` instead (two hosts tried: `8.8.8.8:53`, then `www.microsoft.com:80`). Today every network caller passes an explicit timeout, so this never actually fired — don't reintroduce the global.
- **WinPE awareness**: code checks `os.environ.get('SystemDrive') == 'X:'` to detect WinPE and redirects large temp downloads/extracts to `C:\DV_Temp` instead of the RAM-disk `X:` — this redirect must not depend on any other condition (e.g. whether an offline target OS is selected).
- **Packaging / auto-updater** (`app/gui/updater.py` + `app/common.py`): `resource_path()` resolves `ui.html` via `sys._MEIPASS` when frozen. `check_for_updates` fetches `driver_tool.py` from `raw.githubusercontent.com`, regexes `BUILD_NUMBER`, and if newer downloads and replaces `dist/DriverVarazslo.exe` — that exe is intentionally committed (`git add -f`) so the updater has something to fetch. raw.githubusercontent.com sits behind a Fastly CDN whose edge cache can serve a stale copy for minutes after a push — the `?t=<timestamp>` query only defeats client-side caching, so `check_for_updates` retries 3× with 3s pauses; testing the updater immediately after a push can still legitimately report "no update". Besides the automatic startup check there is a **hidden manual trigger**: clicking the sidebar "DRIVERVARÁZSLÓ" logo runs `checkForUpdatesManual()` (ui.html) — deliberately no visual hint and completely silent when up-to-date; only shows the standard update-confirm dialog when a newer build exists. Never let any edit lower the committed `BUILD_NUMBER` — updates are offered only on strict `new_build > BUILD_NUMBER` (a regression silently freezes updated users; happened once — see `bump_build.py`). All network downloads (`check_for_updates`, `perform_update`, `_download_stresstools`, `_download_block_script`, `bump_build.py`) use `ssl.create_default_context()` with verification **on** — never add `check_hostname = False`/`CERT_NONE`; these paths download admin-executed payloads. On a *brand-new* Windows install the root-cert store is sparse (only schannel clients trigger on-demand root fetching; Python's OpenSSL doesn't), so Python gets `CERTIFICATE_VERIFY_FAILED` for `github.com` (Sectigo/USERTrust root) while `raw.githubusercontent.com` (DigiCert) works. `_download_stresstools` and `_download_block_script` therefore fall back, on that specific error only, to PowerShell `Invoke-WebRequest` (schannel, verification still fully on; side effect: heals the machine's root store). This fallback is NOT a verification bypass and must not be turned into one.

### AutoFix — the 3-leg reboot chain

- **AutoFix resume-after-reboot** (`app/gui/autofix.py`): 3-leg chain. Leg A (fresh start) shows the confirm dialog, pauses WU, schedules a task with `--resume-step1`, reboots. Leg B (a brand new process relaunched by that task) does ghost-device/driver deletion, schedules `--resume-autofix`, reboots. Leg C does the WU rescan/install, looping while new drivers keep installing. **Every reboot uses `shutdown /r /t 0 /f`** — without `/f` the unattended chain can hang forever. **Any Leg-A dialog choice that must affect Leg B cannot be a function parameter** (different OS process): append it as a flag to the scheduled-task `-Argument` string and parse it from `sys.argv` in `GuiBaseMixin.__init__` (worked example: `skip_printer_drivers`/`AUTOFIX_PRINTER_SKIP_CLASSES` in `app/wu_core.py`). The scheduled-task PowerShell is an f-string with single-quoted values — always pass exe path/args through `_ps_quote()` or an apostrophe in a username breaks the chain. Further chain mechanics that are easy to break unknowingly:
  - **The resume legs are started by the frontend, not by Python**: the scheduled task only launches the exe with the flag; `GuiBaseMixin.__init__` parses it into `self.resume_mode`/`self.resume_step1`, `get_init_data()` returns both, and **`ui.html` (in the `pywebviewready` handler) calls `startAutoFix(true)` after a 1s delay** when either is set — that's the only trigger. Consequences: (a) anything that stops the GUI from reaching `pywebviewready` (missing/broken WebView2 → CLI fallback, the single-instance mutex, a dismissed UAC prompt) silently ends the chain with no error anywhere except the debug log; (b) the auto-update check is deliberately skipped on resume legs (the `else` branch) so an update can't hijack a running chain; (c) `startAutoFix(true)` passes `skipConfirm`, so no dialog appears — the printer choice comes from argv, not from the JS `skipPrinters` value.
  - The task is `Register-ScheduledTask -TaskName "DriverVarazsloResume" -Force`, trigger `-AtLogOn`, principal `-LogonType Interactive -RunLevel Highest` (interactive+elevated is required — the GUI must be visible and admin).
  - **Leg B must never leave the machine without a scheduled task while it deletes drivers.** It used to *unregister* the task first thing and only register the next one at the very end — everything in between (restore point, ghost removal, the whole third-party delete loop, minutes of work) was an unprotected window. Field-proven (2026-07, Win10 + R9 200): the machine went down mid-delete with 9 packages already gone; no task existed, so nothing resumed, and the user found a half-stripped machine and had to restart the whole chain by hand. Leg B now **re-registers the task to `--resume-autofix` as its first action** (`-Force` overwrites) — deliberately pointing at the *next* leg, not a retry of itself: if the deletion is what killed the machine, retrying would boot-loop, whereas the install leg continues sensibly from whatever state the deletion reached.
  - **The install leg (C) arms the same safety task**, for the same reason — it is the longest and riskiest stretch of the chain (storage/chipset drivers, the poisoned-session territory), and it too used to unregister the task on entry. It is bounded by an `install_leg_starts` counter in the stats file that counts *every* entry (planned reboots and crash resumes alike); above `AUTOFIX_MAX_INSTALL_LEGS + 1` it stops arming, so a machine that never heals reboots at most once more than a normal chain and then stops. The counter dies with the stats file at the end of the chain (and in leg A).
  - **That safety task must be removed on the error/cancel path** (the `except` in `run_autofix`'s worker unregisters it): it exists only for an *unexpected* death. Without that, a user who stops the process — or any leg that raises — would find the machine resuming into the install leg at the next logon.
  - **Running from `%TEMP%` is handled explicitly**: if `_app_exe_path()` is inside `%TEMP%`, leg A copies itself to `%PUBLIC%\DriverVarazslo_Resume.exe` (`.py` from source) and schedules *that* — otherwise the app's own temp-cleanup / Windows can delete the exe between reboots and the task launches nothing.
  - **Every reboot in the chain goes through `_reboot_or_cancel(status)`** — it keeps the 5s grace window but polls `_cancel_flag` twice a second, and on cancel it **unregisters `DriverVarazsloResume` before raising** the cancel exception. The old `time.sleep(5)` + unconditional `shutdown` ignored a Cancel pressed inside that window (field-seen, Build 224: the user cancelled and the machine rebooted anyway), and without the unregister the chain would have resumed at the next logon regardless — a Cancel that only *looks* like it stopped anything. Do not re-inline a bare `shutdown` at a chain step.
  - **A requested update that never gets a `FOUND:` line must be reported** (`wu_core.unoffered_requested_titles`, used by the AutoFix rounds and the manual `_install_wu_api_sync`): the script walks its *own* search results and emits `SKIP:` for everything not in our filter, so **counting SKIP lines is meaningless** — the real signal is a requested title with no `FOUND:`. Field-seen (Build 224): "Telepítendő driverek száma: 3" followed by `TOTAL: 2`, and the third package vanished without a word (the server had started reporting it as installed after the previous round).
  - **The Store-app sync at the very end is a *bounded* synchronous wait — 10 min, then move on** (explicit user decision, Build 228, reversing the earlier fire-and-forget design): `Invoke-CimMethod -MethodName UpdateScanMethod` runs a full Store scan (measured 112s in the field) that updates the DCH companion apps (Intel GCC, Realtek Audio Console), never drivers. It *used* to be a detached `subprocess.Popen` with no wait — but the chain's final leg does NOT reboot, so the user eventually closes the app, and the exit-time `cleanup_zombies()` `taskkill /F /T` kills the whole process tree, so the fire-and-forget child usually never finished. It now runs via `self._run(..., timeout=600)`: `_run` swallows `TimeoutExpired` → `CMD_TIMEOUT_RETURNCODE`, so on the (rare) 10-min overrun we print "the Windows background will continue it" and proceed, otherwise "done". Don't revert it to a detached Popen — the taskkill-on-exit is why that silently lost the sync.
  - **The install leg re-chains itself** (this predates the 2026-07 fixes but is easy to miss): when the WU round loop ends it reboots and schedules another `--resume-autofix` leg if anything installed **or** if the round was cut short by the poisoned-session guard (`self._autofix_reboot_pending`, set by `_scan_and_install_wu_sync`) — that is how the packages skipped by the `'reboot'` abort actually get installed, on a clean boot. `AUTOFIX_MAX_INSTALL_LEGS` (10, counted from the `legs` array in `autofix_stats.json`, which survives reboots) is a **high last-resort backstop**, NOT the normal stop: per explicit user decision (Build 228) the chain runs "until a round installs zero new drivers, and *that* is done" — reaching the leg cap only happens on a pathological machine where a fresh package genuinely re-installs every leg. It was raised from 3 to 10 once the real Build-228 cause (an already-staged catalog package counted as "1 installed" every leg) was fixed in `_install_catalog_sync` (the "already exists" no-op above); the catalog closing round is also deferred when a reboot is pending. All three scheduling sites go through **`_schedule_autofix_resume(resume_flag)`** — do not re-inline the `Register-ScheduledTask` block, it used to exist in three copies.
  - **The wedged-PnP delete rule (leg B → leg C hand-off, 2026-07)**: deleting a package can wedge the device stack so that *every following* `pnputil /delete-driver` blocks for **~143s** (Windows' own internal query-remove timeout) — field case: an Intel RST `iastorhsa_ext.inf` extension, which turns a 1-minute delete phase into 45+ minutes. **Every** stalled package (`drivers_core.delete_stalled` — our `DELETE_DRIVER_TIMEOUT`=60s firing, or codes 480/482) goes onto the `pending_deletes` list in `autofix_stats.json`, even when the loop carries on — without that the package is simply never deleted *and* WU then sees it as installed, so it never gets replaced either (field-seen on Build 224: `iastorhsa_ext.inf` silently survived a full chain). On the first stall **plus** a pending reboot, `_delete_third_party_sync` also stops trying, adds the untouched remainder, and returns `'wedged'`. **It must NOT reboot there** (explicit user decision — the delete phase runs in one go, then the leg's normal reboot happens): the leftovers are swept at the *start of leg C* by `_finish_pending_deletes()`, where a fresh boot makes each one take ~0.5s (field-measured). That sweep only touches packages named in the list whose *original* INF name still matches (so a renumbered `oemXX.inf` can never take out a freshly installed driver). **Deletion repeats until it is actually finished** (explicit user requirement): if the sweep wedges again it re-defers the remainder and chains another reboot, but only while it is *making progress* — a round that deletes **zero** packages means they are undeletable (e.g. an in-use printer INF) and no reboot can help, so it gives up immediately; `AUTOFIX_MAX_DELETE_ROUNDS` (3) is the backstop. **When the sweep does finish the deletion, the leg reboots once more before installing anything** — the fresh boot both completes the removals and clears the pending-reboot state that would otherwise send the very first install straight into the `[kód=4]` wall. **`DELETE_DRIVER_TIMEOUT` must stay well below 143s** or it never fires — the first attempt used 180s and was inert.
  - Leg A also gates on `_check_internet()` before doing anything (no net → abort with a toast, nothing is deleted), clears `autofix_stats.json`, and calls `_disable_sleep_sync()`; leg B repeats `_disable_sleep_sync()` and runs `powercfg /h off` (Fast Startup off — with hiberboot on, the "reboot" doesn't fully reinitialize the PnP stack and re-enumeration after driver deletion is unreliable).

### What actually gets a driver — the inventory

Four independent sources feed drivers onto a machine, and it is easy to misattribute a gap to the wrong one. **Both the manual scan and AutoFix use the same functions for all of it** — the only difference is that the manual scan lists candidates with checkboxes and AutoFix installs them unattended. Never fork a second implementation for one of the two.

| Source | Covers | Behaviour |
|---|---|---|
| **WU Agent** (`_search_wu_api` + `_match_wu_updates_to_devices`) | every present device, **no class whitelist** — chipset, GPU, audio, LAN, Bluetooth, mice, controllers, cameras | the main path; AutoFix installs automatically, manual scan lists |
| **Update Catalog — error-coded devices** (`_catalog_search_collect`) | devices with `ConfigManagerErrorCode` ≠ 0 that WU had nothing for | automatic in both paths |
| **Update Catalog — generic-driver devices** (`mark_generic_replace_candidates`) | devices running on an inbox driver, in the class whitelist only | manual scan: pre-selected with a badge; AutoFix: closing round |
| **Vendor sites** | NVIDIA (download + silent install), AMD/Intel/motherboard/OEM (link-out only) | end of `start_hw_scan`, fail-silent |

Deliberately **never** driverized, and why — do not "complete" this list without re-reading the reasons:

- **Storage controllers** (`SCSIAdapter`, `HDC`) and the disk itself (`DiskDrive`, already in `WU_SCAN_IGNORED_CLASSES`): a bad controller swap surfaces as `INACCESSIBLE_BOOT_DEVICE` at the *next* boot, where no rollback of ours can run; the gain over Microsoft's inbox NVMe/AHCI driver is ~nil on modern hardware; and this project already has a field case of a vendor storage driver (Intel RST `iastorhsa_ext.inf`) wedging the PnP stack for ~143s per package. What actually helps an SSD is a **firmware** update, which is out of scope. (Explicit user decision, 2026-07-24.)
- **Display, from the Catalog**: the NVIDIA/AMD/Intel cards are a strictly better source — the Catalog is always older.
- **Monitor**: a monitor INF carries EDID/colour metadata, not a functional driver.
- **Keyboard/mouse/HID**: vendors ship *software* (Synapse etc.), not drivers; where a real WHQL package exists the WU path already installs it (field-confirmed: Razer `rz00b9dev.inf`).
- **USB controllers/hubs**: Microsoft's driver is the correct one on modern chipsets, and a failed swap takes out the keyboard and mouse — i.e. the very input needed to recover. (Adjustable if ever needed: drop `USB` from `GENERIC_REPLACE_BLOCKED_CLASSES`.)
- **PCI bridges / root ports / ACPI nodes** (`pci.inf`, `machine.inf`, `hdaudbus.inf`, …): no vendor driver exists at all — see `GENERIC_REPLACE_BLOCKED_INFS`.
- **Printer, PrintQueue, AudioEndpoint, Battery, Processor, Computer, Volume, CDROM, WPD**: excluded at the device-filter level (`WU_SCAN_IGNORED_CLASSES`).

**Asymmetry worth remembering**: the AutoFix *delete* phase removes third-party packages regardless of class, but the *reinstall* only reaches classes that survive `WU_SCAN_IGNORED_CLASSES`. A deleted **printer** driver therefore never comes back through us — which is exactly why the printer checkbox exists and why leaving it on is the shop-safe default.

### WU + Catalog engine (shared by AutoFix and the manual scan)

- **WU driver scan/install — shared core (`app/wu_core.py`)**: the PnP query+filter (`WU_PNP_QUERY_PS` + `_filter_wu_scan_devices`), WU↔device matching (`_match_wu_updates_to_devices` → `_hwid_matches`: **prefix match OR same-bus token-subset**, plus title↔name fallback, without which SoftwareComponent packages never match. The token rule is not a nicety: Windows composes hardware IDs as `VEN&DEV&SUBSYS&REV`, while WU frequently ships `VEN&DEV&REV` — SUBSYS sits *between* the tokens, so neither string is a prefix of the other. Field-proven (2026-07, Win10 + Radeon R9 200): `pci\ven_1002&dev_6811&rev_00` never matched `PCI\VEN_1002&DEV_6811&SUBSYS_30001682&REV_00`, so AutoFix **deleted the vendor GPU driver and silently never reinstalled it** — the machine finished the "successful" run on the Microsoft Basic Display Adapter, error-code-free, so even the summary looked clean. Class-scoped `&cc_` package IDs were unreachable for the same reason. IDs with no bus prefix — bare `usbmmidd` — stay exact/prefix-only, since anything looser false-positives on them), and the WUA installer PowerShell (`_build_wu_install_ps`) exist in exactly ONE copy, used by all three consumers: GUI manual install (`app/gui/hwscan.py`: `start_hw_scan` pool + `_install_wu_api`, filtering by exact `UpdateID`s with HWID-prefix fallback), GUI AutoFix (`_scan_and_install_wu_sync`), CLI AutoFix (FÁZIS 6, `match_system_devices=True`). **Never copy this logic back into a class** — the duplicated era produced the "AutoFix works, manual install silently broken for a whole release" bug (Build ~192). Baked-in rules: downloads are **synchronous `$DL.Download()`** — never `BeginDownload($null,...)` (null callbacks throw NullReferenceException instantly; that was the Build-192 regression; hang protection is the Python-side cancel `taskkill`); selection filters by exact `UpdateID` because distinct WU packages can share one HWID (field-seen: 3 selected → 4 installed with HWID-only filtering); with no filter args the script installs NOTHING (`EMPTY`). The script emits a line protocol (`INIT/SEARCH/FOUND/SKIP/TOTAL/DLONE/INSTONE/OK/OKRB/FAIL/EMPTY/DONE/ERROR`) parsed by all three callers — adding a marker means updating all three reader loops (`OKRB` = success but `$IR.RebootRequired`; counted as success everywhere, surfaced as a reboot warning in the manual GUI path via `self._wu_reboot_required`). **All three reader loops go through `wu_core._iter_process_lines`** (reader thread + queue): cancel is checked every 0.5s even when the script emits nothing, and a watchdog taskkills the process after 30 min of silence (`WuProcessAborted('cancel'|'hang')`) — the old direct `for line in process.stdout` blocked forever (dead Cancel button) when the in-script WUA search hung; don't revert to direct iteration. **The "poisoned session" rules (2026-07, field-diagnosed twice on a Dell OptiPlex 7060 — Builds 214 and 218) — all three live in `wu_core` and are used by both AutoFix paths:**
  - `is_reboot_pending(run)` (+`PENDING_REBOOT_PS`): once a *storage/chipset* driver installs (field case: Intel RST `iaAHCIC`), Windows goes pending-reboot and the WUA then returns `orcFailed(4)` for **every remaining package in that session — after a fixed ~143s wait each**. 8 leftovers = ~20 wasted minutes per round. `_install_abort_reason(consecutive_failures, reboot_pending)` is the shared decision: `'reboot'` if pending, else `'failstreak'` after `WU_MAX_CONSECUTIVE_FAILURES` (3) consecutive install failures (the net for a session that rots without setting the flags), else None. It is wired in through the new `abort_check=` parameter of `_iter_process_lines`, which kills the installer process and raises `WuProcessAborted(reason)` — callers never touch the PID.
  - **The pending-reboot query runs only after an install FAILURE, never after `OKRB`** — deliberate: in the field log the OKRB'd Display driver was followed by three more packages that installed fine, so bailing on mere reboot-*need* would sacrifice good installs. Failure is the evidence; reboot-need is not.
  - `verify_failed_installs(failed_titles, pkgs_before, pkgs_after)`: **a `[kód=4]` is not proof of failure.** All 8 "failed" packages in the field log were sitting in the DriverStore afterwards — the WUA verdict was a false negative. After every round the third-party package list is re-read and any failed title whose version shows up among the newly added packages is counted as a **success** (and never added to `install_failed_uids`). Never report a WUA install failure to the user without this check.
  - `_filter_wu_older_duplicates(matches, wu_by_uid)`: WU offers a package's whole history (field: 10 `iigd_ext` Extension packages for one Intel UHD 630, from 2018), and AutoFix used to install every one of them. Keeps only the newest per `(HardwareID, DriverClass, DriverProvider, DriverModel)` family. That key is deliberately narrow: a wrong merge means a needed driver never installs, while a missed merge only restores the old behaviour.
- `_filter_wu_downgrades` (+`_iso_date_or_none`) is the AutoFix **downgrade guard**: WU can offer a package *older* than the installed driver (field-relevant right after a vendor NVIDIA install); matches whose `DriverVerDate` is strictly older than the installed driver's date are skipped — but NEVER for error-coded devices (any driver beats none), and **NEVER when the currently installed driver is an inbox (Windows built-in) one** (`_is_inbox_driver`: published INF is not `oemNN.inf`, or provider is Microsoft). That second exception is not optional: AutoFix deletes the vendor package, Windows immediately drops its own generic driver on the device, and an inbox driver's date is always newer (it ships with the OS) — so a pure date comparison locks the vendor driver out *forever*. Field-proven (2026-07, X670): `atihdwt6.inf` (AMD HDMI audio, 2021-07-13) was skipped in all three rounds because the inbox `hdaudio.inf` replacing it reports 2026-05-16, and the machine ended the run with one fewer driver than it started with. This is why `_get_installed_driver_info` also selects `DriverProviderName` + `InfName`. The manual scan doesn't filter, it only flags (`downgrade: true` pool items render an orange warning and are not pre-selected) — using the same inbox exception.
- **Printer protection 2.0 (`app/wu_core.py`: `_collect_printer_protection` + `_is_printer_protected`, used by BOTH AutoFix paths)**: class-based skipping (`AUTOFIX_PRINTER_SKIP_CLASSES`) alone was NOT bulletproof — multifunction printer packages spread across USB/Ports/SYSTEM classes (field-seen: `mvusbews.inf`, `hppscnd.inf`, `hpbuio70l.inf`), and deleting those broke customer printers/scanners. Before third-party deletion the code now also protects (a) every INF actually used by present printing components (`Get-PrinterDriver` InfPath + present Printer/PrintQueue/Image devices' `DEVPKEY_Device_DriverInfPath`) and (b) every package whose provider matches a printer vendor present on the machine (`PRINTER_VENDOR_KEYWORDS`; deliberately over-protective — WU restores e.g. HP laptop drivers anyway, a broken customer printer is unrecoverable). **INF matching must check both the published (`oemXX.inf`) and original (`hpc1320u.inf`) names** — `Get-PrinterDriver` reports original names, PnP `DriverInfPath` reports published ones (live-verified on the shop machine).
- **NIC safety net (`app/wu_core.py`: `_export_net_driver_backup`/`_restore_net_driver_backup`, both AutoFix paths)**: AutoFix deletes the LAN/Wi-Fi driver too, and if neither inbox nor WU covers the NIC (field-seen: fresh AM5 + Realtek 2.5GbE), the machine strands offline while the chain needs internet. Before deletion, Net-class packages are exported (`pnputil /export-driver`) to `<app_data>\netdrv_backup`; on the resume leg (GUI) / after install (CLI), if `_check_internet()` fails the backup is reinstalled + rescan.
- **NIC rescue pack (`app/nicpack_core.py` + `app/gui/nicpack.py`/`app/cli/nicpack.py`)**: solves the no-NIC-driver → no-internet → can't-download bootstrap. `nicpack.zip` (hand-built: common LAN/Wi-Fi INFs exported via `pnputil /export-driver`; NOT in the repo — a GitHub release asset like stresstools.zip) is searched next to the exe (service USB!) → app-data dir → downloaded as last resort (same fresh-Windows cert fallback as block.bat). Installs everything via `pnputil /add-driver /subdirs /install` — pnputil only accepts signed, applicable INFs, so blanket-install is safe.

### Vendor-site driver sources (GPU + motherboard)

- **Vendor-latest NVIDIA driver check (`app/gui/nvidia.py`)**: WU's GPU drivers lag months behind; after each manual scan `_check_nvidia_driver()` queries NVIDIA's public services (lookup XML TypeID=3 cached 7 days in app-data → `AjaxDriverService DriverManualLookup`, DCH first then standard; osID 135 for Win11 build≥22000 else 57) for the *real* PCI NVIDIA GPU (`VEN_10DE` — virtual displays excluded), converts installed `DriverVersion` `32.0.15.6636` → NVIDIA `566.36` (last-5-digits rule), and emits `nvidia_driver_info` for the UI card; `install_nvidia_driver()` downloads with MB progress and runs the installer silently (`-s -noreboot`, returncode 1 = ok-but-reboot; cancel works during download only — never kill a GPU installer mid-flight). Product matching: exact name else longest-substring (so "RTX 3060" doesn't shadow "RTX 3060 Ti"); everything is fail-silent so a scan can never break on NVIDIA's servers. Live-verified: GT 710→475.14, RTX 3060→610.74.
- **AMD/Intel vendor check (`app/gui/vendorgpu.py`) + OEM card (`app/gui/oemdrivers.py`) — LINK-OUT ONLY, deliberately**: unlike NVIDIA, AMD/Intel have no stable public download API, and both block referer-less direct exe downloads — so these cards compare versions and open the official page (`open_vendor_driver_page`, https-only guard), they must NOT be "upgraded" to auto-download scraped installers. AMD's latest comes from the *official* GPUOpen `amd-vulkan-versions` repo `amdversions.xml` (cached 3 days; its `<windows-version>` e.g. `32.0.31021.5001` compares directly against `Win32_VideoController.DriverVersion`; newest WHQL by release-date). Intel's comes from the stable-ID download page 785597 (highest `\d+.0.101.\d+` in the page text) and is offered **only** for the modern unified driver family (installed version matches `^\d+\.0\.101\.` or an Arc card) — legacy iGPUs can't apply it. The OEM card deep-links Dell by Service Tag (`.../servicetag/<tag>/drivers`), Lenovo by serial search, HP generic page + serial shown for manual entry. **When `Win32_ComputerSystem.Manufacturer` is a placeholder (`To Be Filled By O.E.M.`, `Default string`, …) — i.e. a self-built machine, which is a large share of shop intake — it falls back to `Win32_BaseBoard` (`BOARD_VENDOR_PAGES`: ASRock/ASUS/MSI/GIGABYTE/Biostar) and shows the board's support page + the board model to type into their search.** That fallback exists because the vendor chipset/audio/LAN drivers for desktop boards are only available there: WU does not offer them, so on a self-built box the onboard audio stays on Microsoft's generic `hdaudio.inf` and the NIC on the inbox driver (field-confirmed 2026-07-24 on an ASRock B450M Pro4 — both still generic after a full, error-free AutoFix chain), and before this the card was simply skipped, leaving the tech no pointer at all. Board links are deliberately the vendor's **support/download landing page**, not a model deep-link (those rot, and MSI/Gigabyte sit behind bot protection) — same reasoning as the HP branch. Only `_is_placeholder`-filtered values are trusted; an unknown board vendor still emits nothing. The card payload carries `title`/`desc` from Python (`renderOemCard` in ui.html falls back to the old wording) because the OEM copy ("gép", "dokkoló") is wrong for a motherboard. All three run at the end of `start_hw_scan`, fail-silent like the NVIDIA check. (A full OEM *catalog* integration — Dell CatalogPC.cab / Lenovo per-MT XML with silent installs — was consciously deferred: blind-shipping untested silent OEM installers to customer machines is riskier than the link-out.)

### DriverStore duplicate cleanup & boot repair

- **DriverStore duplicate cleanup (`app/gui/dupdrivers.py`; the GUI panel lives in the *Temp Fájlok Törlése* view since 2026-07 — it used to sit in the Drivers view, whose freed spot now holds the AutoFix quick-launch hero card)**: groups third-party packages by *original* INF name, keeps the highest version per group, offers the rest for deletion. Safety rules baked in: published INFs actively used by present devices (`Win32_PnPSignedDriver` `InfName` set) are never deletable (shown locked); if that query fails the whole delete is refused (`active_infs is None` → everything marked active); only `oemXX.inf` names are ever deleted; delete tries `pnputil /delete-driver` plain first, `/force` only on failure; the delete worker re-checks the active set at delete time (UI state can be stale). **Automatic closing cleanup**: `dupdrivers_core.auto_cleanup_duplicates(run, log, get_drivers)` is the unattended twin (same safety rules, fail-silent — a cleanup error must never fail the install) and runs at the end of every driver install: GUI manual install (`install_selected_wu`, live system only), GUI AutoFix final leg (right before `_emit_autofix_summary`), CLI AutoFix (after FÁZIS 6, only if something installed).
- **BCD button = BootFixer.cmd downloader (`app/gui/bcd.py`)**: per explicit user decision the "BCD Boot Hiba Javítása" GUI button no longer runs the built-in bcdboot/bootrec repair — it downloads the user's own `BootFixer.cmd` (raw.githubusercontent.com/egonixaimgod/boot_javito_tool/main/BootFixer.cmd, same fresh-Windows cert fallback as block.bat) into `_app_data_dir()` and the UI explains how to run it manually; the app NEVER executes it. ### Manual scan pipeline

- **Manual scan — hybrid pipeline (`app/gui/hwscan.py`)**: the scan now (1) surveys installed driver versions+dates once (`_get_installed_driver_info`, returns `{PNP: {'version','date'}}`) and shows "telepítve: X · date" next to every offer (offer side also shows `wu_date` from `DriverVerDate`/catalog row); (2) after a successful WU search runs a **catalog top-up** for error-coded devices WU returned nothing for — the pool is therefore MIXED (WU items carry `update_id`, catalog items carry `url`), so `install_selected_wu` is a **per-item dispatcher**: one `_safe_thread` worker creates a restore point, then calls `_install_wu_api_sync` then `_install_catalog_sync` (both return `(success, fail, cancelled)` and must NOT emit `task_start`/`task_complete` — the dispatcher owns the lifecycle), and on success emits `offer_rescan` (UI banner offering a new scan — chipset/USB drivers surface new devices); (3) emits a `problems` list (devices with `ConfigManagerErrorCode` ≠ 0, code descriptions in `PNP_ERROR_CODE_DESCRIPTIONS`, `has_fix` flag, `pnp_id`) rendered as a red section in the UI so unresolved devices are never silent — rows with quick-fixable codes get a button calling `fix_problem_device(pnp_id, code)` (22 → `Enable-PnpDevice`, others → disable+enable cycle; synchronous, deliberately bypasses `_task_busy`). The old catalog install path downloaded every cab TWICE (a vestigial sequential loop + the threaded worker) — the sequential loop was removed; don't reintroduce it.
- **Manual scan — WU-API timeout policy & Catalog fallback (`app/gui/hwscan.py`)**: `_search_wu_api` runs the WUA search with a 300s timeout; on timeout it restarts the WU services (that heals the *next* search) but deliberately does **NOT** retry the search itself — field-proven (cloned Windows on brand-new AM5 hardware, 2026-07, twice in a row) that the post-restart retry times out identically, doubling the wait to ~10.5 min for zero benefit. Callers handle the `None` return on their own: manual scan falls back to the Catalog, AutoFix breaks out of the WU rounds (and still runs its catalog closing round). The **Catalog engine** (`_catalog_fetch_rows`/`_catalog_find_driver`/`_catalog_search_collect`; `_catalog_search` is the pool-appending wrapper) scrapes `catalog.update.microsoft.com` on 10 threads, and per device queries **all of its HWIDs** (max 4) plus the synthesized **base vendor HWID** (`wu_core.base_vendor_hwid` — strips `&SUBSYS_`/`&REV_`/`&CC_`), then **unions every returned row into one candidate set** before choosing. Both halves of that are load-bearing and were measured (2026-07-24, Realtek ALC892): (a) the old code stopped at the *first* HWID that returned rows and its comment claimed a more generic HWID "would return the same" — false, the SUBSYS-scoped key returns a 2021 package (6.0.9136.1) while the base key returns the 2026 one (6.0.9992.1); (b) the device's own `HardwareID` list contained **only** SUBSYS-scoped IDs, so without synthesizing the base key that newer package is unreachable entirely. Rows are **scored for the host OS/arch** (`_catalog_row_score` on the tag-stripped `<tr>` text: arm64 rows excluded on x64, Win11-only rows excluded on Win10, Server rows deprioritized; if every row is excluded it falls back to all rows), and among the best-scoring rows the **newest `Last Updated` date wins, with version only as the tiebreak** — *not* highest version: one vendor ships mutually incomparable version schemes for the same device (measured on the Realtek NIC: `Realtek - Net - 1168.19.704.2024` from 2024-07-03 vs `Realtek Net Driver Update (10.79.50.1003)` from 2025-10-02 — by version the 2024 package "wins" by three orders of magnitude). `wu_core._parse_driver_version` (tiebreak only) counts only ≥3-component number sequences as versions, so "2.5GbE" doesn't parse as 2.5; row order on the catalog page is NOT newest-first; it lives in `wu_core` because the AutoFix duplicate/verification filters use it too. Results **whose version is not higher than the device's currently installed driver are skipped** (`_get_installed_driver_info`) — without this the fallback re-offered a driver installed 3 minutes earlier, and this same filter is what makes the AutoFix catalog closing round terminate (the WU API path doesn't need it: the server applies `IsInstalled=0`). Install (`_install_catalog_sync(pool, task_id=...)` — task_id lets AutoFix reuse it on the 'autofix' channel) goes cab-download → `expand` → INF-presence check → `pnputil /add-driver /install` (or `dism /Add-Driver` for offline target). **A pnputil "success" is only a real install when something *new* actually landed**: `Added driver packages: 0` (nothing applied) AND the "already exists in the system" case (every `added successfully` line is a re-add of a package already staged in a previous leg — `Added driver packages: N` with N>0 but all marked `(Already exists in the system)`) are both treated as **no-ops** (counted as skipped, never success). The second check is not optional — a generic-replace candidate that stages fine but never *binds* (field: Realtek UAD audio stays on `hdaudio.inf`) would otherwise report "1 installed" every leg forever, an infinite reboot loop (field-seen, Build 228, capped only by `AUTOFIX_MAX_INSTALL_LEGS`). With the no-op detection the candidate installs once, and on the next leg its package is already in the store → 0 installs → the chain terminates on its own. `.msu` links go through `wusa /quiet /norestart` (offline: `dism /Add-Package`), `.exe` links are skipped with a clear message (never silently run an unknown installer). pnputil still enforces signature + INF applicability, but catalog matching is inherently less precise than WUA's server-side evaluation: it's the B-plan for machines where the WU Agent is broken, not full parity.
- **"Vendor driver instead of the Windows generic one" (`wu_core.mark_generic_replace_candidates` + the `generic_replace` flag; used by BOTH the manual scan and AutoFix)**: the WU Agent never offers a replacement for a device Windows considers fine, so a machine can finish a flawless AutoFix chain with its onboard audio on Microsoft's generic `hdaudio.inf` and its NIC on the inbox driver — while the chip vendor's own package sits in the Microsoft Update Catalog (measured 2026-07-24, ASRock B450M Pro4: Realtek MEDIA 6.0.9992.1 and Realtek Net 10.79.50.1003 both present in the Catalog, zero offers from the WU Agent). Devices whose current driver is inbox (`_is_inbox_driver`) are therefore marked `generic_ok` and fed to the Catalog engine alongside the error-coded ones — the manual scan lists them with a `🏭 most Windows alapdriver → gyári elérhető` badge (pre-selected, the tech can uncheck), the AutoFix closing round installs them automatically. Rules that must not be softened:
  - **For a marked device the version comparison is SKIPPED entirely.** An inbox driver's version is the Windows build (`10.0.26100.8457`) and the vendor's is its own scheme (`6.0.9992.1`), so the generic always looks "newer" and the better driver would always lose. Same trap the WU path already handles in `_filter_wu_downgrades`. This cannot loop: after a successful install the device runs on `oemNN.inf` with a vendor provider, so `is_generic_replace_candidate` no longer marks it (and `added_zero` already guards the AutoFix reboot loop).
  - **Candidate selection is narrow by design** — class whitelist (`GENERIC_REPLACE_CLASSES`), class blocklist, an INF blocklist for Windows *bus enumerators* (`GENERIC_REPLACE_BLOCKED_INFS`: `pci.inf`, `machine.inf`, `hdaudbus.inf`, …), and a vendor-coded `PCI/HDAUDIO/USB` HWID requirement. Without the INF blocklist the rule ballooned from 2 real candidates to 19 on the test machine (17 PCI bridges/root ports on `pci.inf`/`machine.inf`, where no vendor driver exists at all). **Storage controllers (`SCSIAdapter`/`HDC`/`DiskDrive`) are permanently excluded**: a bad swap there surfaces as `INACCESSIBLE_BOOT_DEVICE` at the *next* boot, when the rollback below can no longer run. Display is excluded too — the NVIDIA/AMD/Intel cards already deliver newer vendor drivers than the Catalog.
  - **Every replacement is verified and rolled back if it fails** (`_verify_generic_replacements`, runs after the batch): the published `oemNN.inf` names are captured from the pnputil output, the devices' `ConfigManagerErrorCode` is re-read once the PnP stack settles, and any device that went error-coded gets its freshly installed package deleted + a rescan, at which point Windows re-binds the inbox driver automatically. A device that vanished from the present-device list is reported but **not** rolled back (typically an unplugged USB device — don't discard a good driver over a query miss). This "try it, verify, revert" net is only sound because the inbox driver never leaves the machine, which is exactly why the storage exclusion above is not negotiable.

### AutoFix extras & reporting

- **AutoFix extras on top of the shared core (`app/gui/autofix.py`)**: per-UID `attempt_counts` allows ONE retry of a failed install before permanent exclusion (successes drop out server-side via `IsInstalled=0`; the old first-offer-permanent-exclusion turned one transient download error into a permanently skipped driver); after the WU rounds a **catalog closing round** runs `_catalog_search_collect` + `_install_catalog_sync(task_id='autofix')` on still-error-coded devices (the AutoFix twin of the manual hybrid top-up — before this, a device WU had nothing for stayed broken even when the Catalog had a driver); the no-internet rescue tries the saved Net-driver backup first, then the **local** nicpack.zip (`nicpack_core._find_nicpack_zip` — never the download path, there's no net); per-leg install counts persist in `<app_data>\autofix_stats.json` (legs are separate processes) and the final leg emits a summary (`_emit_autofix_summary`: chain-total installs + **packages that never came back** + remaining error-coded devices) then deletes the stats file (leg A also clears it so a stale chain can't pollute the next run). **The "never came back" report (`_emit_missing_packages`) is mandatory, not cosmetic**: AutoFix deletes every third-party package assuming WU will restore whatever the machine really uses, and that assumption is sometimes false — field-seen (2026-07) `vigembus.inf` (ViGEmBus, a driver installed by a *user application*, which WU does not carry at all) and `amdafd.inf` (its `pci\ven_1002&cc_040300` generic class HWID never matches a present device, so the pairing can't find it) both vanished permanently and silently. We cannot reinstall them (there is no source), but the shop must know they're gone. The delete phase writes the pre-deletion package list to `pre_packages` in the stats file, and the comparison keys on the **original** INF name (`oemNN.inf` gets renumbered on reinstall). Read `pre_packages` *before* `_autofix_stats_total_and_clear()` — that call deletes the file.
  - **Deleting EVERY third-party package is the whole point of AutoFix — never "improve" it into selective deletion** (explicit user decision, 2026-07). This is a repair-shop tool run on machines whose drivers are already bad; the design is *strip everything, let WU put back clean current drivers*. So a package disappearing is NOT a bug to fix by protecting the package, and no "skip drivers that look important / are in use / have no WU match" filter may be added. The only opt-out is the printer checkbox (see Printer protection 2.0), which the user drives. The single legitimate follow-up is **reporting**: `_emit_missing_packages` must state the two real causes — (1) **the device was not plugged in during the fix**, so WU could not offer a driver for it (WU and `_match_wu_updates_to_devices` only ever match *present* hardware: a disconnected BT dongle / controller / printer loses its driver and cannot get it back automatically), (2) it was an application-installed driver WU never carries — and must tell the shop the actual remedy: plug the device in, then run a scan in "Driver Keresés és Telepítés". Field-confirmed on the 2026-07-24 run (`rtkfilter.inf` Realtek Bluetooth + `sie_usb_device.inf` Sony, both for unplugged devices); the old wording blamed WU ("a Windows Update nem ismeri") and offered only "reinstall the program", which is wrong for case (1).
  - **Failed (non-stalled) package deletions are reported too** (`_delete_third_party_sync` collects them via `drivers_core.delete_succeeded` and lists them): a `pnputil` refusal like "at least one device is currently using the specified INF" used to be dropped on the floor while the phase still printed a clean `✅ Driverek eltávolítva` — the same silent-false-success class of bug as the Build 218 `0xC0000142` case. It is not an error (the chain carries on), it just must not be invisible.
  - **`🎉 MINDEN LÉPÉS KÉSZ!` is only emitted when no further leg is chained** (it now reads `EZ A KÖR KÉSZ!`): it used to print at the end of every install leg, immediately followed by a reboot, which reads as "finished" plus an unexplained restart.

### Background task concurrency

- **Background task concurrency (`DriverToolApi` only, `app/gui/base.py`)**: long tasks run via `self._safe_thread(task, target)` — a single `self._task_busy` gate rejects a second concurrent task (shared state: `_cancel_flag`, `hw_updates_pool`), and `_cancel_flag` is reset *only after* the busy-check passes (never reset it in the calling method — a rejected call would clear a cancel-in-progress). `start_hw_scan()` uses its own `_hw_scanning` flag + emits `hw_scan_result` on every exit path, but still participates in `_task_busy`. Individual stress-tool launches bypass `_task_busy` deliberately (narrower `_stresstools_download_lock` guards the shared-ZIP step); `load_drivers()` also bypasses (harmless read-only refresh); `stop_stress_tests()` bypasses (must work mid-task).

### Stress-test tools

- **Stress-test tools — download & power lock** (`app/gui/stress.py`, constants in `app/stress_defs.py`): tools ship in one externally-hosted `stresstools.zip` (GitHub release asset, not in repo — whether a tool launches depends on the ZIP's contents; 2026-07 contents also include CPU-Z, GPU-Z, ZenTimings, NVIDIA Profile Inspector as individually-launchable service tools). `_download_stresstools(progress=None)` downloads+extracts once (marker file; `_stresstools_download_lock` serializes; cache freshness is `_stresstools_cache_valid` — marker + SumatraPDF + HP driver + the *newest* ZIP addition (currently nvinspector) so fielded old caches force a re-download; the optional `progress(phase, done, total)` callback feeds the modal bar in bulk/install mode and the in-view `stress_dl_progress` bar — throttled emitter: `_stress_dl_progress_emitter` — for individual launches; byte-level progress comes from `common.download_with_cert_fallback(progress_cb=...)`, the PS-fallback path has none → indeterminate; a failed download deletes the partial zip — field-seen `[Errno 28]` on a nearly-full C: where the leftover would eat the space the retry needs). `STRESS_TOOLS[key][1]` filename entries may be fnmatch patterns (`gpu-z*.exe` — GPU-Z's exe carries its version number). `start_stress_tests()` ("Stress teszt indítása") launches only `STRESS_TOOLS_BULK` and is the ONLY automated path (dialog automation + 4-quadrant layout). **Individual launches (`start_stress_tool`) have NO automation by explicit user decision (2026-07)**: no click sequences, no Linpack console script, no HWiNFO INI (a previously auto-written SensorsOnly INI is even removed if byte-identical to ours), no window positioning — the user configures the tool manually; if the tool was installed via `toolsinstall.py` the installed copy launches (no download), else portable from the temp package. Per-tool launches skip the progress-modal system (toasts + in-view download bar only, explicit product requirement). Power lock: bulk always; individual launches only for `STRESS_POWER_LOCK_KEYS` (load generators + monitors — CPU-Z/GPU-Z/ZenTimings/Profile Inspector must not disable sleep). Bulk path calls `_lock_power_for_stress()` first (disables monitor/standby/hibernate timeouts AC+DC; original values parsed from `powercfg /query` *positionally* by `0x...` hex — labels are localized, hex order isn't — and saved to `HKLM\<STRESS_POWER_REG_KEY>` first time only), restored on next app startup (`_restore_power_after_stress()` in `__init__`) or by `stop_stress_tests()` (red button only in the stress modal's completed/error state, deliberately NOT permanent in the view) which tree-kills recorded PIDs (`taskkill /T`), sweeps `STRESS_KILL_IMAGES` by name as safety net, then restores power.
- **Stress-test tools — admin-elevated launches (HWiNFO64)** (`app/gui/stress.py`): `_launch_stress_exe` tries `subprocess.Popen`; on `winerror == 740` falls back to `ShellExecuteExW` + `SEE_MASK_NOCLOSEPROCESS` (struct in `app/win32.py`) to get a real hProcess → PID. **Do not "simplify" to plain `ShellExecuteW`** — it returns no PID, which used to silently no-op HWiNFO64's automation/positioning/kill (field-diagnosed via debug log). If ShellExecuteExW returns no handle (UAC dismissed), automation is skipped for that run — the name-based kill sweep exists for that case.
- **Stress-test tools — unattended input automation** (`app/gui/stress_automation.py`, sequences in `app/stress_defs.py`); per-tool Win32 techniques discovered on real hardware — do not regress:
  - *Linpack*: **`SendInput`-simulated real keystrokes** into its console — NOT `stdin=PIPE` (broke launch with `CREATE_NEW_CONSOLE`). **Prompt-verified**: `_read_console_screen()` (AttachConsole + `CONOUT$` + `ReadConsoleOutputCharacterW`) polls until the expected prompt substring appears (60s/step) — never type blind on a timer. `Linpack.exe` is a Bat2Exe-wrapped `.bat`: menus read with `choice` (ONE key, no Enter), run-count with `set /p` (line + Enter) — hence the per-step needs-Enter flag in `LINPACK_PROMPT_SCRIPT`; a stray Enter after a `choice` menu leaves CR in the buffer, the next `set /p` reads empty, and the batch dies silently ~1s after the RAM answer (the long-mysterious field crash). v1.1.8 flow: `2` → RAM option → `10000` → `Y` → `N` (app manages power) → `N` (HWiNFO runs) → Enter. RAM option from `_pick_linpack_ram_option()`: largest option fitting *total* RAM (`_get_ram_gb()`, rounded up) **and** *available* RAM minus ~1.5GB headroom (total-only pick page-thrashed a 12GB box). `SetForegroundWindow` is verified via `GetForegroundWindow()` re-read — Windows silently refuses foreground-stealing.
  - *FurMark / Prime95 / HWiNFO64*: **`BM_CLICK` via `PostMessageW`** — not coordinate clicks (DPI/position-immune, no focus needed) and deliberately NOT `SendMessageW` (blocks until the handler returns; FurMark's button opens a *modal* CAUTION dialog → the "GO" step could never run; field log: stuck 50s). Sequence steps support localized label alternatives (`['Indítás', 'Start']`), `skip_if_found` (Prime95's GIMPS welcome only on first launch — `prime.txt` `StressTester=1` afterwards), `optional`+`timeout` (HWiNFO's post-Start update notifier, dismissed via `Bezárás`/`Close`; `HWiNFO64.INI` `CheckForUpdate=0` suppresses it at source, the step is the net), and `exact` (short labels like `OK`/`Igen` would substring-match Hungarian `...ok` text). `_normalize_ctrl_text()` strips accelerator `&` **and collapses whitespace runs** — real texts include `L&3` and `Just  &Stress Testing` (double space); each alone broke Prime95 in the field. Steps are located with `_find_pid_window_with_child_text()` (60s poll), not `_find_window_for_pid()` (largest-window logic misses small dialogs). The *last* step is verified by `_verify_final_click()` (button must disappear; retries with PostMessage then bounded `SendMessageTimeoutW` 3s).
  - *Console apps* (`_find_console_window_for_pid`): console windows belong to hidden `conhost.exe`, so PID-based `EnumWindows` never finds them — AttachConsole/`GetConsoleWindow`/FreeConsole instead; these mutate process-global state, so every call is serialized through `self._console_attach_lock`.
  - **A Linpack prompt's timeout is a *stillness* clock, not a total budget**: `_auto_answer_console` restarts the `STRESS_STEP_TIMEOUT` countdown on every console-screen change (`_console_screen_is_progressing`), with `CONSOLE_PROMPT_MAX_WAIT` (900s) as the absolute ceiling. On a fresh/cloned Win11 24H2+ the Linpack launcher notices WMIC is missing (it became an optional feature) and **installs it itself via DISM** — a percentage bar crawling for minutes, which blew the fixed 180s cap and aborted the automation under a perfectly healthy program (field log, 2026-07: aborted at `[= 37.8% =]`, then the second run completed in 4.4s because WMIC was by then installed). While the screen is redrawing, the program is working — only true stillness counts as a hang. The same field case exposed a second trap: that WMIC notice ends in a bare `Press any key to continue . . .` **before the menu**, so the console isn't slow, it's *blocked* — our expected prompt would never arrive on its own. `_console_is_blocked_on_keypress` detects a keypress-wait marker in the last 3 screen lines that is **not** the prompt we're currently waiting for, and after `CONSOLE_UNBLOCK_AFTER` seconds of stillness sends one Enter (max `CONSOLE_UNBLOCK_MAX` times). Both guards are needed: the stillness clock alone would still have timed out at the pause.
  - **Window layout waits for the real main window** (`_find_main_window_for_pid`, `STRESS_MAIN_WINDOW_WAIT`/`STRESS_SPLASH_TITLE_MARKERS`): HWiNFO64 spends 20–60s detecting sensors while showing only an "Elemzés…" dialog and no visible main window. Plain `_find_window_for_pid` (largest visible titled window) therefore either found nothing at all ("nem található ablak a pozicionáláshoz") or laid out the tiny splash box into the bottom-right quadrant — both field-seen in the same session. Linpack still goes through `_find_console_window_for_pid` (conhost owns its window).
  - Poll timeouts are deliberately long (`STRESS_STEP_TIMEOUT = 180` in `app/stress_defs.py`, shared by dialog search, console-window search and every Linpack prompt) and `start_stress_tests()` staggers launches by 3s — a loaded machine can take ~a minute to show a dialog even on strong hardware (real capture: FurMark GO at 56s — the old 60s cap was nearly exceeded there, and slow dual-core/HDD machines DID exceed it). Err long, not short.
  - **Slow-machine hardening (2026-07)**: (a) Linpack typing is focus-dependent (SendInput goes to the FOREGROUND window) — after every typed answer the console screen is re-read and must CHANGE; if not, focus is re-taken and the answer retyped (max 3 rounds), with a fresh screen re-check before each retype so a slow redraw never causes double input. (b) `_auto_click_sequence`/`_auto_answer_console` return True/False; `_launch_stress_exe(result_sink=, result_key=)` collects per-tool outcomes and the `task_complete` status names the tools whose automation did NOT finish ("Kézi beállítás kellhet: ...") — process-started is no longer reported as full success. (c) The automation-join ceiling is 600s, and if any automation thread is STILL alive at the ceiling, window layout/minimize is SKIPPED entirely (positioning mid-automation steals focus from Linpack typing and minimizes unanswered dialogs — the old fixed-30s-timer field bug).
  - The three long-standing field failures (Prime95 welcome-skip + `&`-matching, FurMark SendMessage deadlock, Linpack v1.1.8 answers + stray Enter) were root-caused with live repros (2026-07) and verified by confirming actual load. For new field reports: grep `STRESSTOOLS-DEBUG` first, prefer adding logging over guessing timing fixes.
- **Stress-test tools — diagnostics**: every Win32 call in the automation path has its return value checked and logged (`_send_unicode_char`/`_send_vk` return whether SendInput injected). `_debug_dump_pid_windows(pid, context)` enumerates every window+child for a PID on any "not found" path. Automation threads run through `_run_automation_safely()` so daemon-thread exceptions log full tracebacks.
- **Stress-test tools — post-launch window layout** (`_position_stress_windows` in `app/gui/stress_automation.py`): runs only after **all automation threads finish** (each `_launch_stress_exe` appends its thread to `thread_sink`; `start_stress_tests` waits with a 240s ceiling + ~3s settle) — a fixed 30s timer used to fire mid-automation. Layout: 4 work-area quadrants (FurMark TL, Prime95 TR, Linpack BL, HWiNFO BR); the 3 non-FurMark windows are resized, FurMark is *moved* at native size (`SWP_NOSIZE` — resizing crops the render surface/FPS counter) and pushed to `HWND_BOTTOM`. `_minimize_other_windows()` then minimizes everything else including the app's own window, per explicit user request.

### Reports, printing & small features

- **System report generation** (`generate_system_report(self, note=None)` in `app/gui/report.py`): HTML via raw f-string concatenation, called synchronously. WMI/smartctl values must go through the inline `g()` (null-safe) and `e()` (HTML-escape) helpers. **GPU VRAM must not be read from `Win32_VideoController.AdapterRAM`** (32-bit UInt32; NVIDIA returns overflow sentinel ≈4GB for bigger cards — field-confirmed on a 12GB RTX 3060): the PowerShell block cross-references `PNPDeviceID` against `HKLM:\SYSTEM\CurrentControlSet\Control\Class\{4d36e968-...}\00XX\MatchingDeviceId` and reads 64-bit `HardwareInformation.qwMemorySize` (GPU-Z technique), falling back to AdapterRAM only without a registry match. The report `<style>` sets `-webkit-print-color-adjust: exact`/`print-color-adjust: exact` on `*` — without it print/PDF drops backgrounds. The `note` param renders as one bordered `<div class="note-line">` per line (padded to ≥4 lines) — **do not** return to a `repeating-linear-gradient` for ruled lines (inconsistent band thickness at fractional print-DPI, field-confirmed). Sets `self._last_report_path` (used by the store printer).
- **Store-printer one-click report print** (`app/gui/storeprint.py`, GUI-only; `STORE_PRINTER_*` constants at the top of the same file): prints the *last generated* report to the Microstore shop printer (`192.168.35.12`). Flow: TCP probe to JetDirect port 9100 (**not** ping) → find existing printer bound to that IP → else resolve driver via `_resolve_store_printer_driver`: (1) reuse the reference printer's (`BOLT hp LaserJet 1320 PCL 6`) driver if present, else (2) stage bundled `hpc1320u.inf` (`HPDriver` folder in stresstools.zip) with `pnputil /add-driver /install` + `Add-PrinterDriver`. Field-proven: `Add-PrinterDriver -Name` does **NOT** download from WU (the wizard's auto-detection is unreachable from PowerShell), and `pnputil`'s exit code is unreliable (5 with "already exists" success text) — judge from text output and capture `Published Name: oemXX.inf` for exact later removal. HTML→PDF via headless `msedge.exe` (`_find_msedge_exe` — a real Edge install, NOT the WebView2 runtime; can be missing) — the subprocess can return *before* the PDF exists (field-seen ~0.6s), so poll up to 10s. Silent print via SumatraPDF from stresstools.zip. **Cleanup runs in a `finally`**: everything *we* added this run (printer/port/staged driver, tracked via `we_added_printer`/`staged_driver_published_name`) is removed on success AND failure (a half-added leftover port broke the next attempt — hence also the individual if-not-exists guards); pre-existing printers/drivers are never touched; cleanup first waits up to 60s for the queue to drain (Sumatra returns before the spooler actually sends).
- **Net Blokkoló Script** (`app/blockscript_core.py`, thin wrappers in `app/gui/blockscript.py` + `app/cli/blockscript.py`): downloads `block.bat` from a GitHub release URL into `_app_data_dir()` — download only, **never executes it**. Deliberately `.bat`, not `.ps1` (exempt from PowerShell execution policy; self-elevates via UAC). Same fresh-Windows cert fallback as stresstools (verification stays on).
- **BitLocker manager** (`app/gui/bitlocker.py`, GUI-only view): status via `Get-BitLockerVolume`, mapped to Hungarian status text + a `color` key — the mapping keyword-matches the *Hungarian* text, so rewording the strings silently breaks the badge colors. `disable_bitlocker` issues `Disable-BitLocker` and reports "decryption started" (background decryption; status re-emitted after 2s). Both refuse offline mode.
- **Temp Fájlok Törlése** (`app/tempclean_core.py` core + thin `clean_temp_files` in both API classes): `_temp_clean_category_defs(sys_drive)` returns `(key, label, [paths], [services_to_stop_first], default_checked)` (user/Windows TEMP + WU cache are the only defaults; DO cache, WER, D3DSCache, CBS logs, crash dumps, INetCache, color profiles are opt-in), `_clean_folder_contents()` deletes contents entry-by-entry (one locked file doesn't abort the rest), `_empty_recycle_bin()` wraps `SHQueryRecycleBinW`+`SHEmptyRecycleBinW`. New category → add to `_temp_clean_category_defs` once. Service-locked categories (WU → `wuauserv`+`bits`, DO → `DoSvc`, color profiles → `Spooler`) stop the service once per batch, restart once after.

## Git / release rules (IMPORTANT)

- **Never commit, push, or release on your own initiative** — not even after finishing a fix the user asked for. Leave changes in the working tree, tell the user they're ready, wait for an explicit "commitold/pushold/adj ki buildet" instruction. (A push to `main` is a de-facto deployment: the auto-updater serves `driver_tool.py` and `dist/DriverVarazslo.exe` straight from the repo, and it cannot be rolled back by lowering the build number.)
- **When the user asks for a release, use `rebuild_verzioszam_novelessel_es_github_pushal.bat`** — don't re-implement its steps by hand. It ends with `pause` — run as `cmd /c rebuild_verzioszam_novelessel_es_github_pushal.bat < nul`. Its `git add .` picks up the whole `app/` package; `git add -f dist/DriverVarazslo.exe` force-adds the gitignored exe.
- `build.bat` (build only, no bump/commit/push) is safe to run anytime for local verification.
- The GitHub remote URL points at the repo's old name (`.../drivervarazslo`); GitHub redirects it — the "This repository moved" notice on push is normal, not an error.

## Build / release

There is no test suite and no `requirements.txt`; the only third-party runtime dependency is `pywebview` (import-guarded in `app/common.py` with a clear exit message). The build machine runs Python 3.12 + PyInstaller 6.17 (the README's "Python 3.14" mention is aspirational). Everything is Windows-only and admin-only: the app self-elevates, and most code paths call `dism`/`pnputil`/`powercfg`/WMI, so they cannot be exercised on a non-Windows host at all — a syntax check (`py_compile`) plus importing `app.gui.api` / `app.cli.api` is the most that can be verified without running the exe on a real machine.

- **Full release** (bump, build, commit, push): `rebuild_verzioszam_novelessel_es_github_pushal.bat` from the repo root. It calls `bump_build.py` first, which increments `BUILD_NUMBER` **in `driver_tool.py`** from `max(local, GitHub-published)` and keeps `version_info.txt` in sync — never hand-edit `BUILD_NUMBER` to a lower value.
- **Manual build only**: `python -m PyInstaller --clean --noconfirm DriverVarazslo.spec` (preferred), or `build.bat` — but `build.bat` ends with `pause`, so from a non-interactive shell it must be `cmd /c build.bat < nul` or it hangs with no visible output. Output: `dist/DriverVarazslo.exe`. The spec's `Analysis(['driver_tool.py'])` follows the `app/` package imports automatically — no `hiddenimports` needed for new `app/` modules as long as they're reachable through normal imports (a module imported only dynamically WOULD need adding). The spec embeds `icon_drivervarazslo.ico` + `version_info.txt` (keep populated/synced — unsigned exe with stale version info trips Defender) and disables UPX for the same reason. The icon is a 6-image multi-resolution ICO (16/32/48/64/128/256 px, 8-bit) since Build 199 — if it's ever regenerated, keep all sizes (the pre-199 icon was a single tiny image). `build.bat` needs `@echo off` + `chcp 65001`.
- **Run from source**: `python driver_tool.py` (GUI, self-elevates via UAC) or `python driver_tool.py --cli`.
- `DriverVarazslo.spec`, `version_info.txt`, and `dist/DriverVarazslo.exe` are intentionally tracked in git — the exe is what the in-app updater downloads.
